**对话 Boris：Claude Code 之父谈 AI 编码的 ROI、Loops 与瓶颈消解**

**Boris 是 Anthropic Claude Code 团队的核心成员，也是 Claude Code 产品的主要推动者之一。在这次对话中，Boris 给出了一个惊人的数据：从去年 11 月开始，他已经用 Claude Code 编写了 100% 的代码——1,700 个 PR、40 万行新增、25 万行删除——而且大部分编码工作是在 iPhone 上完成的。**

更让人意外的是他的数字：自三月份以来，他个人消耗了 80 亿个 Token。但对于一个每天和 AI 编码工具形影不离的人来说，这个数字反而说明了另一个问题——真正的瓶颈已经不是写代码了。

- <strong>编码的瓶颈已经转移</strong>：写代码的问题基本被 AI 解决了，真正的瓶颈依次是 Code Review → 安全审查 → CI 优化。Anthropic 用 Claude Code Review 抓到了 98-99% 的 bug，下一步是安全产品。<br><br>
- <strong>Loops 是 Agent 的下一代抽象</strong>：从手写代码 → Agent 写代码 → Loops（Agent 提示 Agent），每个步骤都像前一步那么重要。Boris 个人约 30% 的代码已经通过 Loops 编写。<br><br>
- <strong>ROI 比成本重要 100 倍</strong>：公司应该先给所有人发 Token（工程师、产品经理、设计师、会计），让大家自由实验找到高 ROI 的使用场景，再从后端控制成本——而不是一开始就设死预算。<br><br>
- <strong>Fable 是里程碑式的跃迁</strong>：Boris 认为 Fable（Claude Opus 4.5 的下一代）的能力提升比 Opus 4.5 还要大，他已经找不到足够难的问题来测试它了。

跟这种级别的工具创造者聊天，最有意思的不是产品功能介绍——那些在发布会和文档里都能看到。真正有意思的是他对「瓶颈」的阶跃式思维：**编码被解决后，瓶颈跳到 Code Review；Code Review 被解决后，瓶颈跳到安全审查；安全审查被解决后，瓶颈跳到 CI 优化。这就像打地鼠游戏，但每次锤子落地都有一个产品接着。**

这背后其实是一个更本质的判断：AI 模型的底层能力还在指数级增长，每次新模型发布都会解决一个现阶段的瓶颈。所以 Boris 说「我们没有一年计划，这个地方变化太快了，你只能抓住然后一点点地玩」。

## 对话实录

**Raytar：** 你今年写了多少行代码？

**Boris：** 多少行代码？我其实提前准备了答案，因为我知道你会问这个问题。1,700 个 PR，新增 40 万行，删除 25 万行。去年我其实删除的比新增的多，今年新增稍微多了一点。Token 统计方面不太准确，因为有时会被删除或受保留期影响，但自三月份以来，我用了 80 亿个 Token。

**Raytar：** 这些代码行都是你自己写的，还是 Claude Code 写的？

**Boris：** 100% 是我的 Claude Code 写的。从去年 11 月 Opus 4.5 开始就一直这样了。

**Raytar：** 你用手机还是笔记本编码？

**Boris：** 现在我是两者混用。说出来你可能不信，但我现在大部分编码是在 iPhone 上完成的。如果你半年前问我「我们会在哪儿写代码」，我绝对想不到是在手机上。但事实就是这样。很疯狂对吧？

**Raytar：** 确实疯狂。很多人都在问关于效率和 ROI 的问题——我们看到 Uber 等公司开始设置年度或月度预算，比如每个工程师 1,500 美元。但 Anthropic 和其他前沿实验室又在推出能力越来越强、但价格也越来越贵的模型。你怎么看待公司在这两者之间的平衡？

**Boris：** 我很幸运每天能和很多公司交流——我们的客户和潜在客户。大致来说，有些公司从成本角度思考，有些从 ROI 角度思考。我认为 ROI 才是正确的框架，因为你不是为了花钱而花钱，你是要投入然后获得回报。

部署 Claude Code 方面，我认为最成功的公司是给所有人发 Token——不仅仅是工程师，还有产品经理、设计师、数据科学家。他们让整个公司去实验，看看创意从哪来。**因为往往不是你预期的那个人带来新产品的想法——可能是一个角落里的会计，或一个你从没听说过的市场部人员——创新往往来自那里。** 不一定是那位最资深的工程师。

所以我认为公司应该让大家去实验，给团队实验的空间。方式是给人们 Token，给他们实验的安全感——让他们觉得「我可以试一些东西，不会被惩罚」。一旦你找到那些内部使用场景，再从后端控制成本，而不是在前端限制。有一个高消耗的场景起飞了，再去优化它。Claude Code 里有按席位控制成本、使用 Advisor 模型、为整个公司切换模型、按部门设置预算等多种方式。所以有很多方法。

我觉得第二个思考维度是这样的——一开始是部署层面，怎么让 Claude Code 用起来。当公司用了一段时间后，你真的需要思考 ROI。ROI 又分两部分：投资和回报。你可能用代码行数的百分比增长来衡量，但我们刚才看到很多人的手举起来说「我的 100% 代码都是 AI 写的」。那当代码 100% 是 AI 写的之后，你怎么衡量回报？这就变得复杂了。

我们在 Anthropic 看到工程师的代码产出从今年年初开始增长了 8 倍。所以要思考的是：第一，让 100% 的代码由 Claude 写；第二，思考每个工程师的代码加速了多少；第三，把这些想法更快地推向市场。当你观察我们的客户，每个人都处于这个采用曲线的不同阶段。

**Raytar：** 开始是编码，然后越来越多人的手举起来说 100% 的代码是 AI 写的。但之后你意识到上下游还有瓶颈——公司最终想要的是交付更多商业价值。所以「解绑」这些想法，让代码更快、更安全、更准确地部署，这应该就是整个行业正在解决的问题。

如果一年前你在做这场炉边谈话，话题会是「怎么用 AI 写更多代码」。但现在下一个问题是：既然代码将由 AI 写，那下一步是什么？这是我们现在所处的一个非常有趣的位置。

Boris 和我以前在 Threads 一起工作过——顺便说一句，Threads 昨天达到了 5 亿月活，恭喜。我想问一个来自我们前同事 Peter 的问题：**Loops 是下一个炒作周期，还是真的？** 你能给不了解 Loops 的人解释一下它是什么，以及你个人怎么用吗？

**Boris：** 现场有谁试过 Loops 或者 Workflows？好吧，没多少人。对于在场的工程师来说，可以这样理解：

两年前我们手写源代码，然后开始过渡到 Agent 写代码，现在我们正在过渡到 **Agent 提示 Agent** 的阶段。那个「Agent 提示 Agent」就是 Loops。

更具体的解释：我的心理模型是——源代码就像编程中的基础语句，Agent 写代码就像函数，而 Loops 就像高阶函数。所以我们在逐层抽象。**Loops 的重要性不亚于从源代码到 Agent 的那一步。** 从 Agent 到下一个东西，Loops 同样重要，同样巨大。

现在的感觉就像一年半前的 Agent——还相当早期。但我们已经在做了：我自己开始通过手写提示词让我的 Agent 去做事，然后这个 Agent 在一个循环中运行，和所有编码者协作。

另一个例子：我读 Threads 看用户反馈，我可以让一个 Agent 持续循环，每五到十分钟读一次反馈，然后提交修复的 PR。所以一年半前我们处于第一步，现在我们在第三步。思考我们所有的工作——工程师和非工程师（设计师、数据科学家、市场人员），都可以被拆解成这种 Loops。

行业正在到达一个临界点，更大比例的代码将作为 Loops 表达，更大比例的工作也是。我个人来说，平均每天大约 30% 的代码是通过 Loops 写的。如果很努力，还能更高。但还不够流畅。

**Raytar：** 如果你问「有多少工程师在用 Loops 写代码」，也许一年后会有更多人的手举起来。你永远比我们超前三个月到六个月。另一个被讨论很多的话题是——为什么大家应该用 Co-Work？

**Boris：** 对于不熟悉的人，Co-Work 是专为非工程师设计的 Claude Code。下载 Claude 桌面版 App，同一个 App 里有聊天、Claude Code 和 Co-Work。Co-Work 底层就是 Claude Code，使用同样的 Claude Agent SDK。我们之所以说它「不适合工程师」，是因为它专为非工程师加了更多护栏。

Co-Work 有一个完整的虚拟机，有相当精密的隔离机制，我们钩入操作系统确保你不会意外删除东西。还有大量针对 Prompt Injection 的保护。

我每天早上开团队站会时，大家会分享做了什么。现在我的 Co-Work 会在浏览器里打开一个电子表格，列出本周所有工作流，然后自动在 Slack 里给每个工程师发消息问状态更新——有时他们的 Claude 会回复，有时工程师自己回复。然后我的 Co-Work 把结果填回电子表格。零设置，你只需要 Co-Work 和 Claude Chrome 扩展。

还有一个更高级的用法：我以前用 Co-Work 订所有差旅，我会说「这是行程，帮我去订机票」，它会打开浏览器，去 Anthropic 内部用的差旅网站，填好表格，订好票。现在我已经把它自动化了：Co-Work 有一个定时任务，每天检查我的邮件和 Google Calendar，如果发现下个活动在不同城市，它就会自动订机票和酒店。我刚去了东京参加 Code With Claude，之前去了伦敦和柏林，它全程帮我订好了所有航班和酒店——多程航班加酒店，全自动。我只需要确认一下就行。

**Raytar：** 现场有多少人试过 Fable？可能不多。很多人只有 Fable 的有限访问权，或者只有三天的试用期。你怎么看？你显然已经在重度使用它了。

**Boris：** 首先关于 Fable 的可用性，从我们的角度看，这是一个误解，我们正在努力尽快解决。

对于 Fable，我的想法是：去年 11 月 Opus 4.5 发布时，大家都在 Twitter 上说模型编码能力有多好——那个跃迁太大了，很多人第一次开始用 Claude 写所有代码。那甚至是我卸载 IDE 的时刻——因为不再需要了。

**我认为 Fable 在模型能力上的跃迁可能比那次还要大。** Fable 有一种「微妙的多维度思考方式」，它像我最聪明的同事——不是一个粗暴的工具，而是真正能理解问题的深度。之前的模型不理解的东西，Fable 能抓住。它有很多微妙之处，会自然地追问「为什么」三次来挖掘问题的根源。

它在调试中非常有用——你提出一个假设，它会追踪那个假设、寻找证据。编码方面，我真的找不到够难的问题来测试它了。每个我给它的问题，它基本上一次就解决了，或者用很少的提示就解决了。真的找不到更难的问题了。

**Raytar：** 所以 Claude Code 现在是 100% 由 Claude Code 编写的吗？

**Boris：** 是的，从去年 11 月开始就是。Co-Work 也是。在 Anthropic 内部，平均 80-90% 的代码由 Claude Code 编写，而且越来越多的团队已经达到 100%。

**Raytar：** 我也听说 Fable 更贵、也慢一些，但聪明得惊人。你是所有任务都用 Fable，还是会降级到 Opus 和 Sonnet？

**Boris：** 我一切用 Fable。永远。

**Raytar：** 是因为 Anthropic 没有预算限制吗？

**Boris：** 我们也考虑 Token 使用量的。Token 不是免费的。每个我们用的 Token，就是没有给客户的 Token。所以在考虑 ROI 时，如果用 Fable 可以将投资减少 50%——比如用 Advisor 模型搭配 Opus，只在需要时调 Fable——有很多优化方式。但在这个技术采用周期的早期阶段，我建议在成本控制上花 50% 的精力确保成本在掌控之中就好了，**把几乎全部精力放在提高回报上。** 因为现在上行空间比下行优化的空间大得多。

**Raytar：** 我们有 107 个问题，显然不能全问完。Andrew 提出的最高票问题是：**Claude Code 在改善协作方面做了什么？目前它感觉像一个单人工作产品。**

**Boris：** 我们在做一些东西，不久会有消息。在那之前，我建议用 MCP 把 Claude Code 接到 Slack、Teams 或 Discord——你们用的协作工具。

**Raytar：** 下一个高票问题是：**既然 Agent 能做大部分编码，工程师应该专注什么？**

**Boris：** 我们做工程师要做很多事情，不只是编码——跟客户交流、想点子、和设计师及 PM 头脑风暴、做数据分析、决定接下来要建什么、跟其他部门对齐。随着时间的推移，模型能把这些做得比我们更好。但在那之前，需要有人去提示模型，去想下一步要做什么。所以这些工作仍然需要人来做。

**Raytar：** 关于 AI 生成代码的爆发：现在大家写代码都这么快了，Anthropic 内部是怎么处理 Code Review 的？

**Boris：** 写代码之后就是 Code Review，这是我们解决的下一个瓶颈。我们为此建了一个产品，叫 Claude Code Review，任何人都可以用。和市面上任何 Code Review 产品不同的是——它贵得多，因为要用大量 Token 来完全自动化 Code Review。当我作为工程师看到一个 PR 时，基本上所有 bug 都已经被抓出来了（98-99%），甚至已经被修复了。下一个问题是：这个 PR 应该存在吗？这是一个好想法吗？

再下一个瓶颈是安全审查。Agent 引入漏洞的方式和人类一样。我们的答案是 Claude Security 产品——每个星期扫描所有代码库，自动发现问题并修复。现在我们已经到了红队测试和渗透测试都抓不到 Claude Security 已经抓到的漏洞的阶段——这在 Opus 4.5 之前是不可能的。

再下一个是优化 CI。我昨天晚上刚做了一件事：注意到 CI 太慢了，就让 Claude Code 看我的 CI 数据、分析实际耗时，然后优化 CI 让它快得多。它使用了一种叫 Dynamic Workflow 的新功能——模型动态编排数十、数百、数千个子 Agent。这本质上是一种新的 Test Time Compute 形式。用了大概几百万个 Token，昨天晚上跑完了。这在过去需要几天、几周甚至一个月的时间来做性能分析。这只是下一个可以用 Claude Code 解决的瓶颈。

**Raytar：** 所以你们的策略就是一个接一个地解决瓶颈？

**Boris：** 对。

**Raytar：** 最后一个问题：**你和你的团队接下来一年的愿景是什么？**

**Boris：** 我们没有一年计划。这个地方变化太快了。就像指数增长曲线一样——你只能抓住，然后一点点地玩。大的方向和我们过去两年一样：**推出最有能力的模型，用其他产品无法让你体验到的方式去体验它。**

我们几年前就意识到：Sonnet 3.5 在编码上是很大的进步，但没有足够多的产品让你充分体验那个进步。所以 Claude Code 就是一种方式——不再有源代码，你直接使用 Agent，这就是体验新模型的方式。

在接下来几个月到一年里，模型在长时间运行的工作上会更强——Claude 已经在这方面遥遥领先，这个差距还会扩大。代码会更安全、更高质量，模型会更对齐。我们会盯着这些新能力，思考要建什么产品来让人们体验它们。

**Raytar：** 大项目编码不是最大的问题，**长期维护**才是。你们怎么处理维护？

**Boris：** 我们正在用 Loops 做维护。比如有一个持续运行的 Loop，它会检查代码库中的架构问题，或找测试套件中不稳定的测试并修复它们，或找重复的抽象可以删除。

**Raytar：** 你会检查这些改动之后再让它运行吗？

**Boris：** 我是第二种——我让它跑 Loop，然后看它提交的 PR。**提示词其实很简单，关键在于用 Workflow。** Workflow 这个词就是魔法——它会投入更多的 Test Time Compute，产出更好的结果。

**Raytar：** Workflows 和 Loops 有什么区别？

**Boris：** 如果传统 Scaling Laws 有三个因子（数据、模型大小、计算），过去两年我们增加了第四个因子：**Test Time Compute**。简单说就是「模型生成多少个 Token」。这是第一种方式——调大输出 Token 数。第二种方式是 Dynamic Workflows——让 Claude 生成一个像程序一样的东西，在虚拟机里运行，编排其他 Claude 来解决一个任务。这是一种全新的 Test Time Compute 形式，我们还在探索中——本质上是让 Claude 启动几十、几百、几千个 Agent 来完成工作。

**Raytar：** **Fable 还解决不了哪些难题？**

**Boris：** 我们的模型还不完美。一个方面是产品感——我仍然比 Fable 想得出更好的产品想法。创意生成还不是它的强项。但写代码已经比我写得更好了，前端设计也比我设计得好。另一个方面是**分布式系统设计**——思考服务架构、数据流、负载因素等。Fable 在这方面还有提升空间。至于这还会持续多久……我猜大概到年底它会相当不错。

**Raytar：** 最后一个问题：**怎么防止工程师变懒，全盘接受 Claude 给出的所有东西？**

**Boris：** 有两个维度。Claude Code 一直有权限提示——每次要运行命令时它会问你是否同意。但时间长了你会发现大家会产生「提示疲劳」——一直点「是」，根本没有真正看命令内容。我们的安全团队发现了这个问题，因为原本「人在环中」的设计是为了提高安全性，但现在大家因为提示疲劳不停地点「是」。

所以这促使我们开发了 **Auto Mode**——Claude Code 的一个新权限模式。现在 Anthropic 内部大部分用户都在用。每个权限提示都由模型根据对话上下文决定「是」或「否」。这不仅比默认「是」更安全，而且少了一个工程师需要做的事——这意味着可以运行超长时间的 Agent，不用一直坐在那点「是」或「否」。

而且 Claude 模型现在已经基本上不受 Prompt Injection 影响了。如果你看我们的 System Card，100 次攻击的成功率大概在 1%，这是行业里最好的。所以我们可以放心地推出 Auto Mode。

第二个是 **Output Styles**。每当有新工程师加入我们团队，我们让他们用 Exploreratory 输出风格。它不会直接帮你做修改，而是会教你怎么做——「第一步，打开这个文件，用这种方式编辑；第二步，运行这个命令……」所以 Output Styles 和用 Claude 本身就是一个非常强大的学习工具，它能帮助工程师理解代码库里发生了什么，特别是在接触新语言时。
