# A Comprehensive Guide to Model Routing

## 什么是模型路由？

模型路由是在每个 AI 请求进入时，智能地选择最佳模型以最低成本处理它，而不是对所有请求使用单一模型。

路由器与 AI 网关不同。网关提供对各 AI 模型的访问入口。随着 Agent 推理支出的爆炸性增长（单个编码 Agent 会话能烧掉过去一个月的预算），智能模型路由正迅速成为现代 AI 基础设施最重要的层之一。

模型路由可以分解为三个核心维度：
1. **模型池**：有哪些模型可供路由？可能是同一家的两个型号（如 Claude Haiku 和 Opus），也可能是跨厂商和开源模型的几十个选项。
2. **信号**：路由器用什么信息做决策？从简单启发式（提示长度、用户等级）到语义分类器，到学习模型质量预测器。
3. **目标**：路由器在优化什么？成本？质量？延迟？加权组合？

不同路由方案在这些维度上有不同的选择，而正确的选择高度依赖工作负载。

## 网关 vs 路由器

作为一个新兴类别，路由经常遭受术语混淆。"网关"和"路由器"经常被混用。两者的区别很简单：

**网关让你能访问模型。路由器决定用哪个。**

网关是统一接口：让你的应用只需调用一个 API，处理跨厂商认证、标准化请求/响应格式、提供统一计费。网关的价值是整合。但网关不决定用哪个模型。

路由器决定何时用哪个模型。它接收请求并动态选择处理它的模型。路由器的价值是优化：你不再为不需要最强模型的步骤付费。

网关和路由器不是竞争类别，生产级 AI 栈两者都需要。

## 为什么路由对 Agent 至关重要

过去一年，路由从只有最先进团队使用的优化杠杆，演变为企业管理编码 Agent 推理支出爆炸的关键基础设施。**单个 Claude Code 会话每分钟消耗 100 万 token，可持续数小时。许多公司的年度编码 Agent 预算在不到一个季度内烧完。**

问题只会更糟：前沿模型越来越贵、越来越强，意味着 Agent 运行更久、每个任务花费更多。同时最便宜的模型也在变好，扩大了纯依赖顶级模型的机会成本。

**因为编码 Agent 的工作负载复杂度差异极大，当一个前沿模型处理所有任务时，你必然在为简单的步骤多付钱来为困难步骤买保险。** 动态匹配模型到每个请求的路由器，让团队能在不牺牲质量的前提下大幅降低成本。

## 智能路由的价值：成本、质量、延迟

**成本。** 路由的主要价值是驱动显著成本节省的同时维持质量。节省范围在 20-95% 之间，取决于使用场景和模型。大多数工作负载包含一个长尾的简单请求，不需要最强模型。把这些请求路由到便宜模型，把贵模型留给真正需要的步骤，就缩小了团队实际支出和最优策略支出之间的差距。

**质量。** 智能路由可以在质量上超越单个模型。虽然前沿模型在某个基准上的综合分数可能接近，但每个模型在子类别上的表现差异很大。通过将每个请求发送到最强模型，路由可以实现 5-20% 的准确率提升。这是异质性模型优势的自然结果。

**延迟。** 小模型可以更快，更强模型可能使用不同的推理努力设置。将延迟敏感的查询路由到能处理它们的最快模型，产生更响应迅速的应用。

## 缓存感知路由

路由决策与提示缓存的交互是路由最重要但最容易被忽视的方面之一。

当 Agent 发送带有大共享前缀（之前消息、系统提示、工具 schema）的请求时，provider 会缓存该前缀，后续命中只收标准输入价格的一小部分（通常约 10%）。**如果路由器把前十轮发给模型 A，下一轮发给模型 B，新模型必须无缓存地重新处理整个上下文，总成本可能超过留在更贵模型上的花费。** 不了解缓存的路由器是在用一个误导性的成本函数做路由。

缓存感知路由将缓存作为路由决策的一级输入。好的路由器应追踪缓存何时真正热、何时因 TTL 过期（通常 5 分钟）、上下文压缩、媒体附件或前缀编辑而失效。它还利用 sub-agent 路由从更窄任务的便宜模型中受益。

**一个在单轮评估中表现很好的路由器，可能在真实编码会话中失去所有纸上节省——因为它没考虑缓存。** 这是 Agent 工作负载路由与聊天路由最主要的区别之一。

## 路由在 Agent 栈中的位置

**会话级路由。** 有些 Agent 会话简单（修 typo、重命名变量），有些复杂（重构模块、调试分布式系统）。会话级路由器决定用什么模型锚定整个会话。

**Sub-agent 级路由。** 编码框架和其他 Agent 系统经常启动 sub-agent 来并行工作或隔离特定子任务。每个 sub-agent 有自己上下文和自己的需求画像。Sub-agent 级路由为每个生成的 worker 选择合适的模型。

**任务级路由。** 规划任务不同于代码生成任务，不同于摘要任务。用户也可能在会话中途切换任务。任务级路由为每个任务选择正确的模型。

**步骤级路由。** 在一个任务内，单个步骤的变化更大。解释工具调用结果与提出修复建议的推理步骤有不同的要求。

## 确定性路由 vs 智能路由

**确定性路由**依赖预定义规则来决定路由到哪个模型，不考虑请求内容。包括静态多模型编排、回退路由和负载均衡。确定性路由一般用于提高**可靠性**和**可预测性**。

**智能路由**通过分析输入本身来做路由决策。路由器检查请求内容——使用分类器、嵌入模型、级联检查或学习策略——并预测池中哪个模型最能满足特定提示的目标。智能路由一般用于提高**成本**和**质量**。

这两种类别是互补的，生产环境需要两者结合：智能路由器做主要决策，确定性方案做失败回退。

## 智能路由的六种技术

**启发式路由。** 基于提示的表面特征（关键词、长度、正则模式）硬编码规则。设置快但脆弱，难以处理语义细微差别，边缘情况容易出问题。

**语义路由。** 决策由设置时嵌入为向量的示例短语定义。运行时，输入提示被嵌入并通过余弦相似度匹配短语向量，最近匹配的路由胜出。快速、可扩展到多条路由、比关键词匹配更健壮。

**LLM 基础路由。** 路由器本身是一个 LLM，读取输入提示并分类到某个类别，然后将原始提示交给与该类别关联的模型。灵活但自相矛盾——因为需要在每个请求的关键路径上增加一次 LLM 调用，这正是团队想消除的代价。

**复杂度分类器路由。** 训练的分类器估计每个输入请求的难度，然后路由到适合该难度级别的模型。快速且便宜，但复杂度是粗糙的信号：两个相同难度的请求可能对完全不同的模型有更好的匹配。

**预测性路由。** 学习模型——在基准、内部评估或生产流量上训练——预测每个候选模型在给定提示上的表现，然后选择最佳预测结果的那个。这是数据驱动的方法，能最好地利用帕累托前沿。

**级联路由。** 请求先发送给最便宜的模型。如果输出足够好（通过置信度分数、二次检查或验证机制判断），则接受并返回。否则升级到更强模型。保守成本并提供自然的质量保障。

在实践中，生产路由器通常是组合式的——级联叠加在预测分类器上，或语义路由加上确定性关键词检查。

## 如何评估模型路由方案

如果你在考虑购买或构建路由器，应该问这些问题：

- **路由器的推荐依据是什么？** 如果只基于请求头、规则或可用性，它是确定性路由器。
- **模型池是什么？** 锁定到一家？覆盖商业和开源模型？添加新模型多快？
- **目标函数是什么？** 能否表达对成本、质量和延迟的偏好？
- **开销是多少？** 路由器本身有延迟和成本。如果每个请求加 1 秒和 $0.10，可能吃掉大部分它创造的价值。
- **供应商中立吗？** 跨供应商中立的路线往往比锁定到单一供应商的更有长期价值。
- **能否处理 Agent 工作负载？** 会话级上下文、工具 schema、提示缓存、长上下文行为都很重要。
- **缓存感知吗？** 对 Agent 工作负载，这是决定性的。
