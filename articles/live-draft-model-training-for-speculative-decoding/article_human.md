# Baseten 实时 Draft 模型训练：接受率提升 20%，受限流量翻倍以上

> Draft 模型（比如 EAGLE-3 和 DFlash）现在已经是加速大语言模型推理的主流方案了，通常能带来 2-3 倍的吞吐量提升和更低的延迟。但问题是，这些 draft 模型怎么跟五花八门的 base 模型和不断变化的流量保持对齐——这是个挺大的工程挑战。Baseten 的工程团队搞了一套分布式训练管道，直接从实时推理中抽 hidden states 做在线训练，不存离线数据。结果呢？已部署场景**中位数接受率提升了 20%**，部分受限流量模式甚至**涨了 100% 以上**。

## 在线训练：解决四个老难题

传统 draft 模型的训练方法有四个让人头疼的地方：

**存储开销大。** 到了 production 规模，把 hidden states 存下来做离线训练基本行不通。Kimi K2 上**单个样本就能超过 2GB**，完整训练可是要几百万个样本的。

**计算瓶颈明显。** 生成 draft 模型需要的 hidden states，碰上超大模型和超长上下文，**成本高得吓人**。

**对齐会漂移。** base 模型做微调或强化学习之后，**draft 模型的接受率常常往下掉**，除非连它一起重新训——结果就成了「训一次，掉一次」的死循环。

**数据合规难搞。** 在零数据留存（ZDR）环境下，**想存数据做离线训练？门儿都没有**。

Baseten 的方案绕开了数据存储这步，直接从推理流里实时捞 hidden states 来训练 draft 模型，**端到端延迟只有一点点增加**，同时模型能持续适配实际流量。

## 架构设计：推理和训练各跑各的

这条训练管道直接建在 Baseten Inference Stack 里面，跟 Baseten Speculation Engine 深度绑在一起。几个设计思路：

**GPU 执行层面**，管道原生对接现有的推理引擎，single-CUDA graphs 和 overlap scheduler 这些性能特性都能用。为了**不让推理过程出现延迟毛刺**，所有网络通信和数据缓冲都扔给了**专用的后台进程**，配合 overlap scheduler 循环上的精细 CUDA 事件同步，保证 hidden states 提取不会卡住主执行线程。

**内存管理层面**，推理端只管发原始迭代数据，聚合全在接收端做。额外内存的用量跟 `max_num_tokens_per_iter` 成正比，**跟** `max_sequence_length` **无关**——也就是说**长上下文推理不会把 GPU 内存挤爆**。

**训练路径层面**，数据加载器和核心训练循环完全解耦，用 `mmap` 支撑的数组把训练数据缓冲到分页内存。整个请求不用在 device 或固定内存里完整展开，**只有马上要进训练循环了才固到内存里**，进一步省 GPU 显存。

## 底层基础设施

Baseten 还用了两个关键框架来撑这套系统：

**UCXX Python 绑定**搞定节点间的大规模张量传输，高效完成异步 RDMA 通信——在几十到几百个节点的集群上，这是绕过 CPU 瓶颈的必备技能。

**Trio 结构化并发框架**用来搭故障恢复路径。在几百个节点的规模下，硬件坏掉或被抢占不是异常，是常态。Trio 帮团队**把故障隔离在小范围内**，个别节点挂了或网络闪断，不会拖垮整个推理链路。有意思的是，Trio 的 guest loop 模式跟 PyTorch 的同步点（比如 `torch.cuda.synchronize()`）配合起来，**不用创建新线程就能跑异步循环**，GIL 争用也降到了最低。

## 一点观察

这次 Baseten 又是一次典型的「infra 驱动性能提升」——没有新模型架构，纯靠工程手段从生产流量里榨更多价值。20% 的中位数接受率提升，意味着这批用户的推理成本直接降了差不多 20%。推理集群规模那么大的今天，**这笔账算下来相当可观**。另外还有一个信号值得留意：这说明 Baseten 的推理平台上已经积累了大量的生产流量，而这些流量本身就是训练数据——**谁手里有推理流量，谁就握住了持续优化的最佳筹码**。
