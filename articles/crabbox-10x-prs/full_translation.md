我过去只能同时管理 2 到 3 个 Claude Code 会话。

但今年 4 月以来，这个数字快速增长，特别是我们设置了 loops 之后。现在任何时候，我都有至少 5 到 10 个会话在并行运行。其中大部分我从未直接输入过指令：它们来自 loops——这些 loops 自动发现 issue、领取任务、验证更改并自主开 PR。

这产生了前所未有的 PR 量。但也带来了一个新问题：每一个 PR 都必须经过审核并部署到真实客户环境中，而每个 PR 都带着破坏某些东西的风险。

所以瓶颈已经转移了。

> 瓶颈不再是写代码。而是把代码合并到代码库中。

这篇文章讲的是 Harness 中解决这个问题的部分，以及 Peter 的新副业项目 Crabbox 是如何解决这个问题的。

## Agent 需要自己的盒子来验证工作

现在常见的做法是让 Agent 启动一个子 Agent，用 Playwright CLI 测试工作并记录证据：一张截图，或一段完整流程的视频，附在 PR 上。这样 Agent 的工作就容易信任和合并。我不需要相信它说的话——我能看到它在工作。

当我只运行 3 到 4 个 Agent 时这没问题。但一旦有了大量并行会话，它很快就出问题了，因为它们最终会在同一个环境中测试，相互冲突。

要验证，每个 Agent 需要让开发服务器真正运行在自己的代码上。即使你给每个 ticket 分配自己的 git worktree，那也只隔离了写代码这一环。在本地多次运行同一个应用是不具备扩展性的：

- 端口通常写死了，出于充分的原因。第二个实例无法启动。
- 一台笔记本电脑只有一个 Docker 守护进程、一个数据库、一个 OS。每个"隔离"的会话实际上在偷偷共享它们。一个 Agent 尝试新的 schema 就能同时破坏所有其他会话。
- 真实的生产环境堆栈也很费 RAM 和 CPU。五个放不下。

唯一能扩展的办法就是不再在单台笔记本上运行所有东西。

> 每个 Agent 应该获得自己在云中的隔离环境：自己的机器、自己的数据库、自己的开发服务器。沙箱们互不接触。

@SToneoneX 实际上为我们 @SuperDesignDev 平台手工打造了一个版本，效果惊人。我们在 Fly.io 上运行：一个 Firecracker VM，里面装着完整堆栈（通过 docker-in-docker 的本地 Supabase、Redis 和开发服务器），从基础镜像启动并配有持久卷。我们加了机上编排器来启动它、一个 CDP 浏览器来驱动它、挂起和恢复功能让一个盒子在约 3 秒内热恢复，以及一个空闲看门狗在 45 分钟后自动关停——所以我们从不为一个被人忘记关的盒子付钱。它解锁了很多可能性。

但它仍然有一个硬伤。要把代码放到盒子上，它从 GitHub 拉取分支（git fetch）。你必须先 push。未提交的工作目录更改完全无法验证。

于是，当 Agent 在盒子上测试、发现 bug、在本地机器上修复后，你就卡住了。本地有一个脏文件，而盒子里只知道已经 push 过的内容。

常规的 commit、push、CI 流程在这里效果不好。你的仓库里塞满了垃圾 commit。而且你也不想每次都得从头重建盒子。

你真正想要的东西很简单。

> 做一个更改，在几秒内重新测试。

这就是我们看到 Peter 的新副业项目 Crabbox 的地方：https://github.com/openclaw/crabbox

## Crabbox 的工作原理

Crabbox 让 Agent 在云中预热一个盒子，从本地工作目录同步脏 diff，然后实时运行测试。三个命令：

1. `crabbox warmup` 启动一个盒子。
2. `crabbox run -- <command>` 运行任意命令，就好像它在本地一样，但实际上在云盒子上执行。每次运行首先从你的机器同步 diff。你甚至不需要 commit。只要文件夹是 git 初始化的，它就会同步所有未提交的更改，然后运行。
3. `crabbox stop` 关闭盒子并删除它。

整个循环就是这样。
