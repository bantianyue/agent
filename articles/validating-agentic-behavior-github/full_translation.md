# Validating agentic behavior when "correct" isn't deterministic

> GitHub Blog | May 6, 2026 | Updated May 26, 2026
> 14 min read

## 文章概要

本文介绍 GitHub Copilot 团队如何解决 Agent 非确定性行为的验证问题。传统测试框架假设"正确的行为是可重复的"，但 Agent（尤其是使用 Computer Use 的 Agent）的行为路径天生不固定——加载屏出现与否、操作顺序不同、网络延迟变化，都可能导致同一个正确结果被标记为失败（假阴性）。

核心贡献：提出基于**支配分析（Dominator Analysis）**的结构化验证框架，将执行轨迹建模为**前缀树接收器（PTA）**，自动区分"必要状态"和"偶然噪声"。仅需 2-10 条成功轨迹即可构建正确性模型，准确率 100%（vs Agent 自评 82.2%）。

## 背景：Agent 验证的信任缺口

假设你维护一个依赖 Copilot Cloud Agent 的 GitHub Actions 流水线。周二构建通过，周三同样的测试失败——但代码没变。

发生了什么？托管运行器上的小网络延迟导致加载屏多持续了几秒。Agent 等待、适应，仍然正确完成任务。但 CI 流水线仍然标记为失败——不是因为任务失败，而是因为执行路径不再匹配录制脚本或断言的时间窗口。

**Agent 没失败，是验证失败了。**

这暴露了三个痛点：
- **假阴性**：任务成功，但测试运行器不能容忍变化
- **脆弱的基础设施**：测试因时间、渲染或环境噪声失败
- **合规陷阱**：结果正确，但 Agent 行为偏离了测试期望

Agent 系统让开发更快，但我们的验证方法仍然僵硬。在确定性软件中，正确性就是匹配特定输入到已知输出。但 Agent 的行为过程是有意非确定性的。**正确性不是关于遵循一组预设步骤，而是"可靠地达成必要结果"。**

## 为什么现有测试方法对 Agent 失效

传统测试工具在执行路径固定时工作良好，但行为分支时开始崩溃。四个常见范式的局限：

- **断言测试**：需要手动编写每个检查，无法处理有效的替代路径
- **录制-回放工具**：对环境噪声极其敏感，微小渲染差异触发假失败
- **视觉回归测试**：孤立比较截图，不理解执行流程或语义含义
- **ML 预言机**：需要数千训练样本，标记行为时无解释性

这些方法共有的结构性假设：**正确性由特定序列的可观察状态定义。** 对 Agent 系统，这个假设不成立。

## 重新定义正确性：必要 vs 可选行为

### 概念的转变

考虑一个使用 Computer Use 在 VS Code 中执行搜索的 Copilot Agent。一次运行中加载屏出现几秒，另一次 UI 立即加载。

传统测试认为这是两种不同结果。但对开发者来说，加载屏是**偶然的**——它不影响任务是否成功。

将 Agent 行为分为三类：
- **必要状态**：必须出现的里程碑
- **可选变化**：加载 spinner、装饰性 UI 变化等
- **收敛路径**：不同步骤序列最终汇合到同一结果

### 从直觉到理论：支配分析

区分"必须有"和"偶然"行为的概念根植于编译器理论中的**支配关系（Dominator Relationships）**。

在控制流图中，节点 A **支配**节点 B，如果从起点到 B 的每条路径都必须经过 A。

通过将支配分析应用于 Agent 执行轨迹，可以自动识别哪些状态是强制的、哪些是可选的、以及不同路径在哪里收敛。

## 将执行建模为图，而非脚本

### 从线性轨迹到结构化图

模型使用**前缀树接收器（PTA）**执行行为：

- **节点**表示可观察状态（UI Agent 的截图 / 开发 Agent 的代码快照）
- **边**表示转换，捕获节点间的动作（点击、按键、API 调用）

### 为什么图很重要

将执行视为图允许表示分支和收敛——线性脚本无法捕获的概念。

- **分支**处理非确定性环境变化（加载屏出现与否）
- **收敛**标识不同路径汇合点，表示 Agent 成功应对变化并返回主流程

## 解决方案：结构化的正确性方法

### 工作流：从轨迹到"主图"

1. **捕获（PTA 构建）**：收集 2-10 条成功执行轨迹，转换为 PTA（有向图，节点=UI 状态，边=动作）

2. **泛化（语义合并）**：将轨迹合并为统一图。使用**三层次等价检测框架**——结合快速视觉指标与 LLM 语义分析——判断两个状态是否逻辑等价

3. **提取骨架（支配分析）**：对合并图应用支配分析，识别"必要状态"——每条成功轨迹必须经过的里程碑，同时自动过滤加载 spinner 等"可选"状态

### 三层次等价检测框架

这是状态等价判断的核心机制：

- **视觉指标**：使用快速感知哈希和 SSIM（结构相似性）立即捕获接近相同的状态
- **LLM 语义分析**：视觉指标模糊时，用多模态 LLM 判断差异是否有语义意义——如忽略时间戳变化但标记不同错误消息或缺失的 UI 控件
- **保守合并**：仅在模型确定等价时合并状态，允许图在路径真正分叉的地方自然分支

这不是朴素的逐像素比较，也不是让 LLM 判断整个任务的"万能胶"。LLM 被防御性地、有节制地用于解决特定歧义。

### 评估结果

在 Copilot Agent 自定义 VS Code 扩展测试套件上的控制实验：

| 指标 | Agent 自评 | 支配树方法 | 提升 |
|------|-----------|-----------|------|
| 准确率 | 82.2% | 100% | +17.8% |
| 精确率 | 83.3% | 100% | +16.7% |
| 召回率 | 60.0% | 100% | +40.0% |
| F1 分数 | 69.8% | 100% | +30.2% |

**关键发现：** Agent（CUA）经常将失败误报为成功（超时或误判状态）。支配树方法通过聚焦于必要里程碑是否真正达到，实现了完美区分。

更关键的是"非 Bug"场景的识别：
- Agent 自评**完全无法（0% F1）**识别"不是 Bug"的场景——Agent 还不能可靠地自评成绩
- 支配树模型在识别"执行错误 vs 产品回归"时达到 **52.2% F1**

## 融入开发工作流

本框架被设计为增强软件开发周期的关键环节：

- **GitHub Actions 流水线**：减少环境噪声导致的假阴性
- **回归测试**：用稳定版本的已验证轨迹创建"真实来源"模型
- **Agent 评估**：独立于 Agent 自评的度量方式
- **UI 自动化**：更鲁棒地处理桌面和 Web 应用中 UI 元素或路径的微变

## 当前限制

- **需要成功轨迹**：2-10 条成功执行，无法仅从失败日志学习
- **LLM 依赖**：语义等价检查依赖多模态 LLM，引入外部 API 依赖和延迟
- **时间盲点**：验证事件顺序，但无法标记某个状态持续过长

## 为什么这很重要

我们不需要黑盒模型去评判其他黑盒模型。我们需要开发者可以审查、推理和信任的**结构性保证**。

通过结合经典编译器理论（支配分析）与多模态 AI，可以从少量示例中学习可解释、鲁棒的成功定义。

对 Agent 验证来说，关键在于将"真实来源"从 Agent 的内部逻辑转移到学习到的外部结构，使自主 Agent 成为现代基础设施中可信的生产级工具。
