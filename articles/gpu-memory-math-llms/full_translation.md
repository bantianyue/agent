# GPU Memory Math for LLMs (2026 Edition) — 完整翻译

如果你在本地运行模型，一旦你考虑到权重最初是如何训练和量化的，"模型→VRAM"的简单对应关系就不成立了。

有一个更好的思考方式：

> VRAM（GB）≈ 参数量（B）×（每权重的有效位数 ÷ 8）

就是这样。

这一个公式能解释所有格式：

- FP16 / BF16
- FP8 / INT8
- GPTQ / AWQ / NF4
- GGUF 变体
- 基本上你用的每种格式

## 你真正需要的唯一换算

核心直觉如下：

- FP16 / BF16 → 16 位 → 每 1B 参数约 2 GB
- FP8 / INT8 → 8 位 → 每 1B 参数约 1 GB
- 4 位量化 → 约 4 位 → 每 1B 参数约 0.5 GB

GGUF 格式介于之间，取决于具体方案：

- Q6_K → 每 1B 约 0.82 GB
- Q5_K → 每 1B 约 0.69 GB
- Q4_K → 每 1B 约 0.56 GB
- Q3_K → 每 1B 约 0.43 GB
- Q2_K → 每 1B 约 0.33 GB

超激进量化可以更低，但有代价。

如果你只记住一件事，记住这个：

- FP16 = 模型大小的 2 倍
- FP8 = 模型大小的 1 倍
- 4 位 = 模型大小的 0.5 倍

其他一切只是这个主题的变体。

## 题外话：没人谈论的显存税

在你考虑权重之前，先明白这一点：模型本身只是你显存账单的一部分。真正的杀手是它周围的一切。KV cache 随上下文长度增长，在 32K、128K 或更高时会悄悄吞噬你的内存。激活值因运行时和优化级别而异，但在某些执行路径下可能飙升。批处理和并发会快速倍增内存使用，尤其是在 Agent 风格的工作负载中。框架开销根据你使用 Transformers、vLLM、TensorRT-LLM 还是 llama.cpp 而不同。还有 CUDA Graphs，它用额外预留内存换取更好的延迟和吞吐量稳定性。底线：如果你只为权重做预算，你已经内存不足了。

## 实际中是什么样

让我们把它翻译成真实的模型大小。

一个 7B 模型：
- FP16 → 约 14 GB
- FP8 → 约 7 GB
- 4 位 → 约 3.5–4 GB

一个 13B 模型：
- FP16 → 约 26 GB
- FP8 → 约 13 GB
- 4 位 → 约 6–7 GB

一个 70B 模型：
- FP16 → 约 140 GB
- FP8 → 约 70 GB
- 4 位 → 约 35–40 GB

一个 405B 模型：
- FP16 → 约 810 GB
- FP8 → 约 405 GB
- 4 位 → 约 200+ GB

现在你明白为什么人们要么：
- 激进量化
- 跨 GPU 分片（如 Tensor Parallelism）
- 或者干脆放弃说"上云吧"

以下是对人们实际拥有的 GPU 的实用换算。

- 约 3B FP16
- 约 6–7B FP8
- 约 12–13B 4 位

12 GB VRAM：
- 约 5B FP16
- 约 10B FP8
- 约 18–20B 4 位

16 GB VRAM：
- 约 7B FP16

- 约 25B 4 位

- 约 10–12B FP16

- 约 20–24B FP16

这是模型权重的"实际能装下"版本。

## 为什么你的模型还是崩

正如我们之前所说，即使数学上算出来装得下，你仍然可能内存不足。

因为权重只是故事的一部分。

你还需要为以下内容预留内存：
- KV cache（长上下文时急剧膨胀）
- 激活值（取决于运行时）
- 批处理 / 并发
- 框架开销

经验法则：
额外加 10–30% 的 VRAM 来安全运行。

如果你在做：
- 长上下文（32K、128K 等）
- 高并发
- Agent 工作流
- ……你还需要更多。

## MoE 陷阱

混合专家模型让人困惑。

例子：
- "8x7B"听起来像 56B
- 但每个 token 只运行一部分专家

所以计算成本 ≠ 内存成本。

重要的是：
- 总参数 → 影响内存占用
- 活跃参数 → 影响速度

取决于模型如何加载：
- 你可能仍然需要所有专家的内存
- 或者你可以将它们分片到多个 GPU

如果你把 MoE 当密集模型处理，你要么严重高估要么严重低估。

## GGUF 不是魔法

GGUF 被当作作弊码来对待。

它不是。

它是一个容器 + 量化策略，针对以下场景优化：
- llama.cpp 风格的推理
- CPU + GPU 混合设置
- 超高效内存使用

但是：
那些内存数字只在该运行时中有效。

一旦你进入其他框架：
- 权重可能被反量化
- 内存使用可能急剧增加

所以"6 GB 装得下"不是普遍真理。它是运行时特定的真理。

## 唯一重要的思维模型

没有你需要记住的巨大的兼容性矩阵。

只有这个：

VRAM ≈ B × (bits ÷ 8)

然后调整：
- 运行时开销
- KV cache
- 并发

就是这样。

一旦你内化了这个，你就不再猜测。

你开始设计系统。

更重要的是，你不再问："我能运行这个吗？"

你开始问："我想怎么运行这个？"

这时候事情才变得有趣。
