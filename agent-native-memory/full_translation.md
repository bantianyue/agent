# Are We Ready For An Agent-Native Memory System? — 完整翻译

**来源：** arXiv 2606.24775v1  
**作者：** Wei Zhou, Xuanhe Zhou, Shaokun Han, Hongming Xu, Guoliang Li, Zhiyu Li, Feiyu Xiong, Fan Wu（上海交通大学、清华大学、MemTensor）  
**代码：** https://github.com/OpenDataBox/MemoryData

## 摘要

LLM Agent 的记忆已经从简单的检索增强机制演变为完整的持久化数据管理系统，支持存储、检索、更新、整合和生命周期治理。但现有的评测将记忆视为一个黑盒，忽视了运营成本、架构权衡和动态更新下的鲁棒性。本文从数据管理的角度提出系统性实验研究，将 Agent 记忆分解为四个核心模块（表示与存储、提取、检索与路由、维护），评估了 12 个代表性系统在 5 种工作负载（11 个数据集）上的表现。核心发现：

- 没有单一架构在所有场景中占优；有效性取决于工作负载瓶颈与记忆结构的对齐程度
- 基于图的方法处理知识更新最可靠
- 许多系统在证据距离拉大时遭受灾难性遗忘
- 局部维护的性价比优于全局重组
- 保守整合是最优的默认维护策略

## 1. 引言

LLM Agent 的快速发展催生了多种记忆架构设计：
- **流式+反思型**（如 MemoryBank）：带时间戳的流 + 周期性反思总结
- **分层分级型**（如 MemGPT/Letta）：多级记忆，支持淘汰/晋升
- **知识图谱型**（如 Mem0^g、Zep）：实体、关系、时间演化
- **混合复合型**（如 A-MEM）：跨多个存储基质的模式感知对象

本文聚焦于**系统级记忆**——与 LLM 权重和上下文窗口解耦的持久化数据管理层，而非任务特定的 Agent 框架。

**现有评测的四点不足：**
1. 未在统一工作负载下覆盖多种代表性架构
2. 依赖单一的端到端任务指标（F1、BLEU），缺乏多维度评估
3. 很少测量运营成本（索引构建时间、查询延迟）
4. 将记忆系统视为黑盒

**本文贡献：**
1. 技术分解与分类法——四个模块的结构化分类
2. 端到端性能评估——5 种工作负载、11 个数据集、12 个系统 + 2 个基线
3. 细粒度组件评估——逐步替换单一模块的控制变量实验
4. 六大核心发现

## 2. 预备知识

### 数据管理视角的 Agent 记忆

- **Agent 记忆（ℳ）**：维护超过单次推理步骤的持久化数据管理对象
- **Agent 记忆系统（ℳ_sys = ⟨ℛ, 𝒮, 𝒬, 𝒰⟩）**：
  - ℛ — 记忆表示与存储
  - 𝒮 — 记忆提取
  - 𝒬 — 记忆检索与路由
  - 𝒰 — 记忆维护

### 与 RAG 和上下文工程的区别

RAG 是**无状态、只读**的检索原语；上下文工程是动态选择上下文窗口内容的实践；而 Agent 记忆系统是**持久化、可更新**的基础设施，管理完整的长期记忆生命周期。

### 与传统数据库工作负载的区别

- 访问是**语义性的**（自然语言、近似匹配）
- 记忆在**持续、可能冲突的观测**下演化
- 工作负载**高度异构**（长上下文合成、情节回忆、结构化事实查找、时间推理、流式更新）

## 3. 方法论：四模块分类法

### 3.1 记忆表示与存储

**逻辑表示：**
- **Token 级序列**：显式离散文本 Token（Mem0、MemoChat）或隐式连续向量 Token（嵌入、KV cache）
- **图/树拓扑**：时间知识图谱（Zep、Mem0^g）或分层树结构（MemTree）
- **异构复合体**：多部件数据容器（MemOS MemCube、LightMem 三部分模式、A-MEM 原子笔记）

**物理存储与索引：**
- **临时上下文寄存器**：无外部存储（MemoChat、MemAgent）
- **专用单引擎**：向量 DB（Mem0、MemTree、Letta），图 DB（Zep、Mem0^g），关系型 SQL（LightMem、Letta），文件/对象存储
- **异构多引擎**：动态构建多种索引类型或分布数据到多个后端（SimpleMem、MemOS、MemoryOS、A-MEM）

### 3.2 记忆提取

- **原始序列拼接**：最小化计算开销，直接拼接对话轮次
- **无模式语义提取**：提取独立的、高价值的信息单元（如 Mem0 提取离散事实语句）
- **模式约束结构化提取**：LLM 填充预定义结构模式，产生类型化数据（Zep、Mem0^g 提取实体-关系三元组）

### 3.3 记忆检索与查询路由

- **原生注意力检索**：利用 Transformer 注意力机制隐式权重和路由
- **语义密集检索**：KNN 搜索连续向量空间（Mem0、LightMem、MemTree）
- **拓扑子图遍历**：沿显式关系边遍历语义簇（Mem0^g、A-MEM）
- **自主 Agentic 路由**：LLM 作为主动查询规划器，生成函数调用或隐式搜索条件
- **多阶段混合执行**：顺序混合路由（先谓词过滤后语义搜索）和并行集成检索（同时多索引查询后融合）

### 3.4 记忆维护

- **时间戳多版本**：用时间戳元数据和 append-only 日志逻辑废弃过期事实
- **容量驱动物理淘汰**：基于约束的硬淘汰（FIFO、Token 限额）和基于分数的优先级淘汰（时间衰减、访问频率）
- **LLM 驱动语义整合**：在线压缩和 CRUD 工具调用
- **连续参数优化**：异步后台过程优化模型参数（如 MemoRAG 的 RLGF）

## 4. 端到端评估

### RQ1：整体有效性

- **LongMemEval**：Zep 领先（48.0 LLM Judge Accuracy），Cognee 35.3 ROUGE-L F1
- **LoCoMo**：MemOS 最佳 EM（11.5），混合过滤最强
- **DB-Bench**：Long Context 48.20 EM，MemoChat 55.40 Task Success Rate

**发现 O1：** 没有单一系统主导所有工作负载。结构感知方法整体最稳健。
**发现 O2：** EM 不足以衡量跨会话综合能力；需要 ROUGE-L / LLM Judge。

**Finding 1（工作负载对齐记忆）：** 强 Agent 记忆不是单一通用表现形式，而是由对工作负载瓶颈的支持程度决定：(1) 分散跨会话推理需要关系和时间感知检索（Zep、Cognee）；(2) 长对话需要粗到细过滤（MemOS、MemoryOS）；(3) 有状态执行需要保留交互轨迹。

### RQ2：检索保真度

- **SimpleMem** 最佳 Recall@1（39.0），但 A-MEM 和 MemTree 在 Recall@5/@10 更强（69.5/85.9 和 59.7/80.5）
- **发现 O3：** 检索是"证据补全"问题。压缩导向记忆擅长早期命中；链接/分层记忆擅长收集分散证据。平面密集检索随时间距离急剧下降。

**Finding 2（以证据为中心的记忆组织）：** 检索质量更多取决于系统如何组织证据供后期重构，而非如何将一条相关记忆排到第一。

### RQ3：更新鲁棒性

- **知识更新**：Zep 领先（44.4 Substring EM, 36.8 ROUGE-L F1）
- **时间推理**：Cognee 领先（18.7 Substring EM, 35.8 ROUGE-L F1）
- **骨干鲁棒性（O5）：** MemOS 在四种 LLM 骨干上表现最稳定。更新行为在最终生成前已基本确定。

**发现 O3（时间更新保真度）：** 更新后行为的可靠性是流水线级设计问题，不是纯模型容量问题。

### RQ4：长视野稳定性

- **LongBench**：SimpleMem 稳定（35.2→34.9），Long Context 骤降（42.6→19.0）
- **LoCoMo**：Embedding RAG 从 37.1 跌至 7.4 Answer F1；图/整合系统保持较高水平

**发现 O4（视野结构化记忆）：** 长视野下主要挑战不是"存储更多历史"，而是"选择正确的抽象层次"。

### RQ5：运营成本

- **LightMem** 和 **MemTree** 处于最强的效率前沿
- 高效用结构化系统移至昂贵侧：Cognee 和 Zep 超过 84 Utility 需要 116.5s 和 155.1s
- **发现 O7（局部维护）：** 性价比最高的机制是将维护限制在有界子集的系统

**Finding 5（运营扩展规则）：** 效率由维护范围而非结构本身决定。

## 5. 细粒度组件比较

### M1：表示与存储
**发现 O8（内容保真度）：** 保留原始会话内容比增加抽象或层级更重要。
- 原始文本在恢复精确会话级细节时最有效
- 轻量压缩仍支持组合推理，但削弱精确匹配
- 更深的分层无法恢复表示中被移除的信息

### M2：记忆提取
**发现 O9（覆盖率保持提取）：** 写时提取中保守、低选择性的提取方案最适合保留下游可回答性所需的上下文。
- 粗粒度主题分组不易断开持续线索
- 轻量记忆化更可能保留需要组合推理的细节
- 同时保留用户和助手轮次有利于理解依赖澄清的对话

### M3：检索与路由
**发现 O10（规划与融合）：** 明确的查询规划和平衡的检索融合能最大程度提升检索效果。
- 适度融合优于稀疏偏向融合
- 明确规划始终优于直接检索
- 规划上加反思不再带来收益

### M4：记忆维护
**发现 O11（保守整合）：** 保守整合比延迟刷新或过度粗粒度的总结更有效。
- 保守合并保留跨轮次关联
- 延迟刷新使近期证据分割化
- 过渡粗粒度的总结掩盖稀疏但有价值的线索

## 6. 结论

本文从数据管理视角系统评估了 Agent 记忆系统，通过端到端评测和细粒度组件消融，为选择合适记忆架构提供了指导。核心发现：

1. **工作负载决定架构选择**，没有通用银弹
2. **图/时间感知记忆**最擅长处理知识更新和时间推理
3. **保留原始内容的表示**优于高度抽象的表示
4. **保守写时提取 + 适度检索融合 + 保守整合维护**是最稳健的组合
5. **局部维护**的性价比远高于全局重组
6. 长视野下**抽象层次选择**比存储容量更关键
