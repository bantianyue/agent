# Cursor 开发者习惯报告：Spring 2026

Cursor 发布首份《开发者习惯报告》，基于平台数据从五个主题描绘了 AI 编程的转型现状：开发速度翻倍、模型经济学分化、头部开发者差距拉大、上下文消耗激增、编程向自动化演进。

## 1. 开发加速

开发者正在更快地上代码，但生产力变化远不止于数量。

AI 正在改变工作的形态——PR 越来越大，Agent 会话越来越深，AI 生成的代码在代码库中存活的时间更长。

**开发者每周添加的代码量正在加速增长。** 自 2026 年初以来，增速持续加快。虽然这不是一个完美的指标，但为理解开发者工作方式的变化提供了一个有方向性的基线。

从 2025 年 1 月到 2026 年 5 月，**每位开发者每周新增代码行数从 3,600 行增长到 8,600 行**——超过两倍的增长。增长曲线在 2025 年上半年相对平缓（3.6K → 4.7K），但在 2026 年初出现明显加速，直冲 10K 以上。

**PR 规模也在扩大。** 中位 PR 行数（P75）同比增长约 2.5 倍——从 2025 年 1 月的约 126 行增长到 2026 年 5 月的 **345 行**。

**"大 PR"（≥1,000 行）占比上升。** 合并 PR 中超过千行的份额从 2025 年 1 月的 8% 增长到 2026 年 5 月的 **13.8%**。一个值得注意的跳跃发生在 2026 年 1 月，当时开发者在测试新的 Agent 改进功能。

**Agent 深度也在增加。** 过去两个月（3-5 月），每次会话的工具调用数增长了约 30%——从 3 月的约 113 次增长到 5 月的 **145 次/会话**，表明 Agent 正在处理更复杂的任务（文件编辑、shell 命令、网页浏览）。

**代码质量/存活率也在提高。** AI 生成代码中，在被接受后 60 分钟仍然保留的份额从 2026 年 1 月的 76% 增长到 5 月的 **81%**。

**一点观察：** PR 变大不完全是好消息。345 行/PR 意味着 review 难度同步上升。AI 生成代码的存活率提升说明质量确实改善了，但代码体积膨胀也可能带来更大的集成风险。关键是——更大的 PR 是否需要更多的人工 review 时间来消化？

## 2. 智能的经济学

模型价格差异很大，但"每条被接受代码行的成本"缩小了高端和低端模型之间的差距。

### 每次 Agent 请求的平均成本

不同模型家族之间的价格差异接近 **9 倍**：

| 模型家族 | 每次请求成本 |
|---------|------------|
| **Opus 4.7** | $1.57 |
| **Opus 4.6** | $0.86 |
| **GPT 5.5** | $0.81 |
| **GPT 5.4** | $0.46 |
| **Sonnet 4.6** | $0.44 |
| **GPT 5.3 Codex** | $0.30 |
| **Composer 2.5** | $0.18 |

### 每条被接受代码行的平均成本

差异缩小到约 **7 倍**。高价模型部分通过"每次请求产生更多被接受的代码"来证明其价格的合理性：

| 模型家族 | 每条被接受代码行成本 |
|---------|------------------|
| **Opus 4.6** | 1.19 分 |
| **Opus 4.7** | 1.10 分 |
| **GPT 5.5** | 1.09 分 |
| **GPT 5.3 Codex** | 0.56 分 |
| **GPT 5.4** | 0.54 分 |
| **Sonnet 4.6** | 0.54 分 |
| **Composer 2.5** | **0.18 分** |

### 成本-质量前沿（CursorBench 3.1）

性能与平均任务成本的对比：

| 模型/配置 | 每个任务平均成本 | CursorBench 3.1 得分 |
|----------|--------------|------------------|
| **Opus 4.7 (Max)** | $11.02 | 64.8% |
| **Opus 4.7 (Extra High)** | $7.11 | 61.6% |
| **Opus 4.7 (High)** | $5.01 | 59.4% |
| **GPT-5.5 (Extra High)** | $4.37 | 64.3% |
| **GPT-5.5 (High)** | $3.59 | 62.6% |
| **Composer 2.5** | **$0.55** | **63.2%** |
| **Sonnet 4.6 (Max)** | $3.09 | 49.0% |
| **Gemini 3.5 Flash** | $1.94 | 49.8% |

**Composer 2.5 以 63.2% 的高分（与 Opus 4.7 Max 仅差 1.6 个百分点）和 $0.55 的极低成本**，代表了成本-质量前沿的重大变化。

**一点观察：** Composer 2.5 的表现值得关注——$0.55 的成本只有 Opus 4.7 Max 的 5%，但 CursorBench 分数仅差 1.6 个百分点。如果这个基准测试足够真实地反映实际编程任务，那么绝大多数开发者应该选择 Composer 2.5，除非任务复杂度超出它的处理范围。

## 3. 权力用户的鸿沟

AI 带来的生产力提升高度集中。前 1% 的开发者正在显著拉开与普通开发者的差距。

### 集中度指标（Gini 系数）

系数越高表示越少用户群体占用了更多资源：

- **AI 生成代码行数：** Gini **0.77**
- **AI 花费：** Gini **0.75**
- **Tokens：** Gini **0.72**

### 产出差距（每位开发者每周新增代码行数）

P90/P99 用户与中位数（P50）之间的差距正在扩大：

| 百分位 | 2025 年 1 月 | 2026 年 5 月 | 增长倍数 |
|-------|-----------|-----------|--------|
| **P50（中位）** | 176 | 712 | 约 4 倍 |
| **P90** | 2,500 | 8,800 | 约 3.5 倍 |

P99 开发者每天产出代码行数是中位数的 **46 倍**，每周合并的 PR 是中位数的 **15 倍**。P90 开发者每天产出是中位数的 **10 倍**，每周合并 PR 是中位数的 **4 倍**。

**一点观察：** Gini 系数 0.77 意味着 AI 生成的代码高度集中在少数用户手中。这不是"AI 让所有人效率翻倍"的叙事——这是"会玩的人已经起飞，还在摸索的人原地踏步"。当 AI 工具降低门槛的同时，也可能让"会用的"和"不会用的"之间产生更大的鸿沟。

## 4. 上下文的崛起

Agent 正在消费越来越多的上下文（输入 tokens）来提高代码质量，正在改变 token 经济。

### 输入/输出 Token 比例

模型正在"写之前读更多"：

- 2026 年 1 月：**4.5 倍**比例
- 2026 年 5 月：**约 11.4 倍**比例

### Token 用量占比（非缓存）

输入 tokens 现在占主导地位：

- 2026 年 1 月：输入 **81.9%** / 输出 **18.1%**
- 2026 年 5 月：输入 **91.9%** / 输出 **8.1%**

### 等值 Token 成本占比

尽管输入 token 的单价更低，但现在它们驱动了大部分成本：

- 2026 年 1 月：输入 **47.5%** / 输出 **52.5%**
- 2026 年 5 月：输入 **69.5%** / 输出 **30.5%**

### 缓存读取主导

缓存读取 tokens 占总活动的主导地位：

- **缓存读取占比：** 持续保持在 **约 89-90%** 的总 token 活动
- **输入/输出占比：** 合并仅占 **约 7-8%** 的总活动
- **缓存写入：** 约 **2.5-6%** 的总活动

**一点观察：** 输入 token 占比从 81.9% 飙升到 91.9%——Agent 正在变得越来越"爱读书"。这意味着模型在写代码之前会先读大量的代码上下文。这提高了代码质量，但也意味着 token 成本结构正在发生深刻变化——输入端的开销可能比很多人预期的要大。

## 5. 向自动化的转变

Agent 正在从个人开发者使用的工具，演变为整个构建和维护软件的系统，甚至经常是自动化的。

### 从工具到系统

编码 Agent 正在经历范式转变：

**从「个人工具」到「系统级基础设施」：** Agent 不再只是帮助一个人写代码，而是开始承担代码库级别的维护任务。这包括自动代码审查、CI/CD 集成、bug 修复、功能开发等完整生命周期。

**从「人工驱动」到「自动化」：** 越来越多的代码变更在不完全人工干预的情况下自动完成。合并 PR、修复 bug、更新依赖——这些以前需要开发者手动操作的任务正在被 Agent 自动化处理。

**从「单次任务」到「持续运维」：** Agent 不再只是完成一次代码生成就结束，而是在代码库上持续工作——监控、修复、迭代。这种持续运维模式正在改变软件开发的工作方式。

### 自动化成熟度

报告指出，自动化正在从早期采用走向主流。当 Agent 能够自主完成完整的代码生命周期时，软件开发的经济模型将发生根本性变化——不再只是"开发者 + 工具"，而是"开发者管理 Agent 团队"。

**一点观察：** 这个转变的核心矛盾在于——当 Agent 越来越擅长自动维护代码库时，"开发者"的角色定义本身就需要重新思考。不是"谁写代码"的问题，而是"谁决定写什么代码"的问题。决策权将比编码能力变得更稀缺。

## 6. 方法论

本报告基于 Cursor 平台数据，对 agentic 编码趋势进行了数据驱动的分析。数据涵盖多个模型家族的实际使用数据，时间跨度从 2025 年 1 月到 2026 年 5 月。

CursorBench 3.1 是一个评估模型在编程任务中表现的基准测试，综合评估代码生成质量和成本效益。
