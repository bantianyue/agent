**My Week with Fable — 实测 Anthropic 新模型 Mythos**

本文作者花了一周时间测试 Anthropic 的 Fable（代号 Mythos），结论是：**它用起来不像任何其他模型，感觉、定价都像下一代模型——但也有真实存在的怪癖。**

**tl;dr**

Fable（Mythos）和其他模型完全不同。它感觉像下一代模型，定价也是。但它也有真实存在的怪癖。

**优点**

**Workflow 模式是最突出的亮点。** 作者让它做一次"完整代码审查"，然后看着它启动了数百个并行 Agent。它给应用中的几乎每个文件都分配了一个独立 Agent，找出了 bug、边界情况、缺失的文档和 UX 改进。他用同样的提示词给其他模型，没有一个能发现这么多问题。

**Fable 极其自主。** 它比任何之前的 Claude 或 GPT 模型都更愿意跑开去连续工作几个小时。最重要的是，作者说他信任它能完成设定的目标。它会心甘情愿地烧掉大量 token 来达成目标。每次启动 Fable，它都像要接手一个大项目。

**作者说他对交给 Fable 大规模复杂任务的信心，超过他用过的任何其他模型。** 他想不出有什么问题能让它卡住。而 Fable 看起来也非常渴望接手这些大任务。

**这就是这个模型的突出之处——长周期任务。** 很难想象它的时间视野极限在哪里。

**不过它也不是神级模型。** 有些地方需要改进。

**怪癖**

**它极其啰嗦。** 解释很快就钻到细节深处。作者更新了他的 claude.md 想约束它，但根本不够。他不断要求它用更简单的方式解释。不只是啰嗦，还有信息密度的问题——它解释问题的方式让他觉得自己真的很笨。

关于信息密度的一点思考——作者说以前没意识到这有多重要。**在固定的 token 预算内能传达更多信息，意味着模型可以以更低的成本变得更聪明。** 这也是未来 Agent 发明自己超密集语言的有力论据。

**Fable 喜欢问澄清问题。** 一个简单的提示词会变成：提问→总结回答→确认总结→写 spec→确认 spec→确认 Agent 方案（并行 vs 串行）→最后才开建。作者希望它自己来做这些决策。Anthropic 告诉他这会在更新系统提示词后修复。

**它感觉很慢。** 比之前的 Opus 模型甚至 GPT 都慢。启动慢，处理问题也慢吞吞。这和作者喜欢 Opus 的原因正好相反——Opus 在原始 token/秒和找到更短解决方案路径上都比 GPT-5.5 快。Fable 在这两方面都不同。即使是简单任务它也慢慢爬。作者看着计时器往上跳，输出 token 却一动不动，五分钟过去只用了几千个 token。**它想尽可能彻底，这需要时间。**

**结论**

**小技巧：把 effort level 调低，比你想象中还要低。** 即使在中等设置下它也会思考很多。即使在低 effort 下它也非常有能力，仍然会思考一段时间。

**所有这些怪癖都是可以修复的**——通过模型优化和增加算力来提速，通过更多微调/RL 和系统提示词调整来解决啰嗦和过度谨慎的问题。

**结论：MYTHOS 极其有能力，作者还在摸索如何发挥它的最大价值。** 它像在要最难的任务，任何不够难的任务都不够好。这是一个全新训练运行的第一次亮相，它已经是他用过的最有能力的模型了。

这是他反复在想的那句话。

**一点观察**

Matthew Berman 的"一周实测"系列的价值在于真实编码场景，而不是 benchmark 分数。但需要知道的是，他的内容风格偏向第一印象和可传播性，而非系统性评测。

**Fable 的 Workflow 模式（数百个并行 Agent 做代码审查）是这个评测里最有信息量的部分。** 这暗示了下一代模型的核心竞争力可能不是单次回答的质量，而是并行任务编排能力。如果模型能同时检查你项目里的每一个文件，那单次对话的"聪明程度"反而成了次要指标。

啰嗦和慢是同一个硬币的两面——模型在花更多时间"思考"，但思考过程暴露给了用户。这引出一个产品设计问题：用户是否愿意为更好的结果接受更慢的响应？Anthropic 用 effort level 调节给了用户选择权，但默认值的选择会决定大众的第一印象。

**"信息密度"那段值得单独拿出来说。** 如果模型能学会用更紧凑的表达传递同样信息，那在相同推理预算下，它本质上就更聪明了。反过来，这也是为什么 Agent 之间可能演化出自己的高效通信语言——不是科幻，是 token 经济学的必然。
