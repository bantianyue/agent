# Baseten 实时 Draft 模型训练：接受率提升 20%，受限流量翻倍以上

> Draft 模型（如 EAGLE-3 和 DFlash）已成为加速大语言模型推理的主流方案，可带来 2-3 倍的吞吐量提升和更低的延迟。然而，让这些 draft 模型与多样化的 base 模型及动态流量保持对齐，始终是一项重大工程挑战。Baseten 的工程团队构建了一套分布式训练管道，直接从实时推理中提取 hidden states 进行在线训练，无需离线存储数据，在已部署场景中带来了**中位数接受率提升 20%** 的效果，部分受限流量模式甚至出现 **100% 以上的提升**。

## 在线训练：消除四大瓶颈

传统 draft 模型训练方法面临四个痛点：

**存储开销巨大。** 在 production 规模下，保存 hidden states 用于离线训练完全不可行。Kimi K2 上的**单个样本就可能超过 2GB**，而完整训练需要数百万个样本。

**计算瓶颈突出。** 生成 draft 模型输入所需的 hidden states 在超大规模模型和超长上下文场景下**成本极其昂贵**。

**对齐持续漂移。** 对 base 模型进行微调或强化学习后，**draft 模型的接受率往往会下降**，除非同时重新训练它，，这形成了一种「训一次、降一次」的恶性循环。

**数据合规难题。** 在零数据留存（ZDR）环境下，**存储数据用于离线训练根本不可能**。

Baseten 的方案绕过了数据存储，直接从推理流中实时提取 hidden states 训练 draft 模型，**端到端延迟增量极小**，同时能让模型持续适应自定义流量。

## 架构设计：让推理和训练并行不悖

这条训练管道直接构建在 Baseten Inference Stack 内部，与 Baseten Speculation Engine 深度集成。关键设计思路有三点：

**GPU 执行层面**，管道原生对接现有的推理引擎，兼容 single-CUDA graphs 和 overlap scheduler 等性能特性。为了**避免推理过程出现延迟尖峰**，所有网络通信和数据缓冲被卸载到**专用后台进程**，配合 overlap scheduler 循环上的精细 CUDA 事件同步，确保 hidden states 提取不会阻塞主执行线程。

**内存管理层面**，推理端发送未经筛选的迭代数据，仅在接收端聚合。增加的内存使用量与 `max_num_tokens_per_iter` 成正比，**而非** `max_sequence_length`，，这意味着**长上下文推理不会被挤出 GPU 内存**。

**训练路径层面**，数据加载器与核心训练循环完全解耦，使用 `mmap` 支持的数组将训练数据缓冲到分页内存。完整的请求无需在 device 或固定内存中实例化，**只在即将进入训练循环时才被固定在内存中**，进一步节省 GPU 显存。

## 底层基础设施

Baseten 还使用了两个关键框架支撑这套系统：

**UCXX Python 绑定**用于处理节点间的大规模张量传输，高效完成异步 RDMA 通信，，在数十到数百个节点的集群中，这是绕过 CPU 瓶颈的必备能力。

**Trio 结构化并发框架**用于构建故障恢复路径。在数百节点规模下，硬件故障和抢占是常态而非异常。Trio 帮助团队**将故障隔离在局部**，防止少数节点宕机或瞬态网络故障导致整个推理链路中断。有趣的是，Trio 的 guest loop 模式与 PyTorch 同步点（如 `torch.cuda.synchronize()`）结合，可以**在不创建新线程的情况下运行异步循环**，从而最小化 GIL 争用。

## 一点观察

这是 Baseten 的一次典型「infra 驱动性能提升」，，不涉及新模型架构，纯靠工程手段从生产流量中榨出更多价值。20% 的中位数接受率提升意味着这批用户的推理成本直接降低约 20%，在推理集群规模巨大的今天，**这笔账算下来相当可观**。另一个值得关注的信号是：这暗示 Baseten 的推理平台上已经沉淀了大量生产流量，而这些流量本身就是训练数据，，**谁拥有推理流量，谁就拥有最好的持续优化能力**。
