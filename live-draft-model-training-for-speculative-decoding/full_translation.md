Draft 模型（如 EAGLE-3 和 DFlash）已成为加速大语言模型推理的广泛采用技术，可带来 2-3 倍的更高吞吐量和更低延迟。然而，让这些 draft 模型与多样的 base 模型及动态流量模式保持对齐，仍然是一项重大的工程挑战。

我们构建了一个分布式训练管道来解决这个问题。该管道直接从实时推理中提取 hidden states，并利用它们实时训练 draft 模型，无需离线存储数据。

在已部署的地方，它带来了 **中位数接受率提升 20%** 的效果，某些受限流量模式甚至出现了 **100% 以上的提升**。这些增益直接转化为更大的投机解码加速和更好的服务效率。

## 消除传统瓶颈

训练 draft 模型的传统方法受到以下几个痛点的制约：

1. **存储开销：** 在 production 规模下，保存 hidden states 用于离线训练是不可行的。Kimi K2 上的单个样本可能超过 2GB，而完整的 draft 训练需要数百万个这样的样本。
2. **计算瓶颈：** 生成 draft 模型输入所需的 hidden states 可能极其昂贵，特别是对于在长上下文下运行的大规模模型而言。
3. **对齐漂移：** 对 base 模型进行微调或强化学习通常会降低 draft 模型的接受率，除非同时对其进行重新训练。
4. **数据合规：** 在零数据留存（ZDR）环境中，存储数据用于离线训练可能很困难。

为了克服上述每一个瓶颈，我们构建了一个分布式训练管道，它直接使用来自推理的实时 hidden states 实时训练 draft 模型，同时为服务路径增加极小的开销。

这种架构完全绕过了数据存储的需求。它显著减少了为新 base 模型训练 draft 模型所需的时间，同时使这些模型能够持续适应自定义流量。

在已部署的地方，我们观察到中位数接受率提升 20%，某些受限流量模式的接受长度提升超过 100%，从而在 production 中实现了更快的投机解码和更高效的工作负载。

## 设计最小开销的 Draft 模型训练

![推理节点和训练节点在分布式集群中并行运行。](https://www.datocms-assets.com/104802/1782399206-speculator-diagram_kb-update_62426-1.png?auto=format&w=1200)

推理节点和训练节点在分布式集群中并行运行。原始数据直接从推理过程流向训练聚合器，消除了数据存储的需求。Hidden states 通过 CUDA-IPC 双缓冲进入经由 mmap 映射的分页内存，保持内存开销最小。该架构在数据并行和上下文并行两个维度上水平扩展，因此 draft 模型训练与实时推理持续并行进行，而不会竞争资源。

### 优化推理路径：GPU 执行、内存和网络

我们将训练管道原生构建在 Baseten Inference Stack 内部，作为 Speculation Engine 的一部分，因此它直接运行在为我们的服务路径提供动力的同一个高度优化的推理引擎之上。这一点至关重要，因为系统需要持续提取训练数据而不减慢推理速度。训练管道与现有的性能特性完全兼容，包括 single-CUDA graphs 和 overlap scheduler。

为了避免推理期间的延迟尖峰，我们将所有网络通信和数据缓冲卸载到一个专用的后台进程。结合 overlap scheduler 循环上的精细 CUDA 事件同步，这使得我们能够持续提取 hidden states 而不会阻塞主执行线程。

为了节省内存，推理端发送未经筛选的迭代数据，仅在接收端进行聚合。增加的空间使用量与 `max_num_tokens_per_iter` 成正比，**而不是** `max_sequence_length`，这为长上下文推理保留了宝贵的空间。

### 优化训练路径

在训练端，我们将数据加载器与核心训练循环完全解耦。管道使用 `mmap` 支持的数组将训练数据直接缓冲到分页内存中。

与推理端类似，增加的 GPU 和固定内存使用量与 `max_num_tokens_per_iter` 成正比，为训练过程保留了宝贵的空间。与其在 device 或固定内存中实例化完整的请求数据，不如在可分页内存中组装完整的请求，仅在即将进入训练循环之前才将其固定。

### 其他基础设施处理工具

我们还想重点介绍在此项目中使用的几个框架：

- **UCXX：** 在节点之间移动大型张量需要专门的网络基础设施。我们使用 UCXX Python 绑定来高效地处理异步 RDMA 传输。
- **Trio（结构化并发）：** 在数十或数百个节点的规模下，硬件故障和抢占是正常运维的一部分。我们使用 Trio 构建了包含这些故障的重试和恢复路径，防止少数节点宕机或瞬态网络故障中断推理。此外，我们将 Trio 的 guest loop 模式与 PyTorch 同步点（如 `torch.cuda.synchronize()`）集成，以运行异步循环而无需创建新线程，从而最小化 GIL 争用。
