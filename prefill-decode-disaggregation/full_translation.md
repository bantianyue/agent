# Prefill/Decode 分离：为什么你的 GPU 不能同时做两件事

自从在 CMU 修了 ML 系统课程以来，我一直在深入研究 LLM 推理领域。现代 LLM 推理有一个奇怪的系统问题：第一个 token 和随后的每个 token 想要的硬件行为完全不同。

乍一听很奇怪。毕竟是同一个模型在生成同一个回复。

但在底层，这两个阶段的行为差异大到把它们强行放在同一个 GPU 上会创造出一个昂贵得惊人的妥协。

Prefill/Decode 分离（disaggregation）正是因这个妥协而存在的。

## 推理的两个阶段（快速回顾）

推理过程中的自回归生成发生在两个阶段：

**Prefill 阶段：** 模型一次性处理所有输入 token。如果你发了一个 1000 token 的 prompt，模型运行一次大的前向传播，同时处理所有 1000 个 token，构建起 KV cache。这是第一次迭代。

**Decoding 阶段：** 模型一次生成一个 token，使用之前所有 token 的 KV cache。这个阶段对你生成的每个 token 都会重复。

这两个阶段感觉相似，但它们是根本不同的工作负载。而这就是整篇文章要说的。

## TTFT 和 TPOT：两个完全不同的问题

你知道那种感觉——你问 ChatGPT 一个问题，然后等待。光标闪烁，什么也没发生，然后突然响应开始流式输出。

第一个字出现前的等待就是 **TTFT（Time To First Token）**，即 prefill 步骤的总耗时。之后文字流出的速度是 **TPOT（Time Per Output Token）**，由 decode 步骤的速度决定。你也可以把它理解为两个输出 token 之间的延迟。

TTFT 慢意味着产品感觉坏了——用户以为什么都没发生。TPOT 慢意味着响应读起来卡顿。两者都以完全不同的方式损害用户体验，并且需要完全不同的优化。

主要问题是：在同一个 GPU 上，优化一个就会伤害另一个。

## 为什么 Prefill 和 Decode 冲突

| 维度 | Prefill | Decode |
|------|---------|--------|
| 瓶颈 | 计算密集型 | 内存密集型 |
| 批处理 | 擅长大规模批处理 | 批量大小有限 |
| 并行化 | 张量并行效果好 | 张量并行效果差 |
| KV cache | 从头构建 | 从缓存读取 |

这些阶段不仅仅是略有不同——它们要的是根本不同的 GPU 行为。

**Prefill 是计算密集型。** 在 prefill 期间，模型一次性处理整个 prompt。例如，如果你的 prompt 有 1000 个 token，所有 1000 个 token 一起流经每个 transformer 层，做大型矩阵乘法。这保持了 GPU 的 tensor core 在执行大量计算时的繁忙，所以瓶颈主要是计算，而不是内存移动。

由于计算量如此之大，张量并行等技术效果很好——跨 GPU 同步结果（如 all-reduce）的通信开销相对于庞大的计算量来说相对较小。

**Decode 是内存密集型。** Prefill 之后，生成变成自回归：模型一次产生一个 token。假设模型生成下一个词，比如"cat"。即使只产生那一个 token，每个 transformer 层仍然要运行一次。

这意味着每个 decode 步骤，GPU 都在反复做：从 HBM（高带宽内存）加载层权重 → 计算 → 移动到下一层 → 重复。

即使单个 token 的计算量很小，模型权重仍然巨大（数十亿参数），所以大量时间花在从 HBM 获取权重上。GPU 花在等待数据上的时间比花在实际计算上的时间还多。

同一个 GPU 有两个完全不同的瓶颈。而且因为瓶颈不同，跨多个 GPU 的并行化方式也完全不同。对 prefill 效果好的方案对 decode 有害，反之亦然。这就是核心矛盾——要正确理解它，我们需要讨论将 LLM 拆分到多个 GPU 的三种主要方式。

## 并行化问题

当你在多个 GPU 上扩展 LLM 推理时，有三种主要策略。每一种对其中一个阶段效果很好，对另一阶段效果很差。

### 张量并行

想象一个 8192×8192 的权重矩阵 W——6700 万个数字。在张量并行中，你把这个矩阵分片到多个 GPU 上。GPU 1 得到左半部分，GPU 2 得到右半部分。

当一次前向传播运行时，每个 GPU 同时将输入乘以其权重的分片。然后它们通过一次 all-reduce 操作通信——每个 GPU 将其部分结果广播给所有其他 GPU，并求和得到最终答案。然后下一层以同样的方式运行。

通信发生在每一层。对于一个 80 层的 70B 模型，每次前向传播有 80 次 all-reduce 操作。

为什么这对 prefill 效果好而对 decode 效果差？对 prefill 来说，你一次处理 1000 个 token。每次 all-reduce 同步的是 1000 个 token 的计算结果。计算量如此之大，以至于通信开销（在 GPU 间广播和求和的时间）只是总时间的一小部分——值得。

对 decode 来说，你一次生成一个 token。每次 all-reduce 同步的是一个 token 的计算结果。计算量极小，但你仍然为每个 token 支付完整的通信成本——每 token 80 次。就像为了分享一句话组织一场 4 人视频通话——建立通话的开销比实际对话还大。

### 流水线并行

与拆分权重矩阵不同，流水线并行拆分模型层。如果你有一个 80 层模型和 4 个 GPU，GPU 1 处理第 1-20 层，GPU 2 处理第 21-40 层，GPU 3 处理第 41-60 层，GPU 4 处理第 61-80 层。

把它想象成工厂装配线。输入进入 GPU 1，经过第 1-20 层处理，然后输出（称为激活值）传递给 GPU 2，GPU 2 处理第 21-40 层，依此类推，直到 GPU 4 产生最终输出。

这里的通信比张量并行简单得多——你只是把单个激活张量从一个 GPU 传递给下一个。没有 all-reduce，没有广播给每个 GPU，就是 GPU 1 交给 GPU 2。

但这听起来很不错，但流水线并行引入了气泡开销。除非流水线被足够的并发工作完全填满，否则一些 GPU 会处于闲置状态。

对 prefill 来说，你可以通过将多个请求放入流水线来填充这些气泡——当 GPU 2 在处理请求 1 时，GPU 1 开始处理请求 2。流水线保持满。但对 decode 来说，你为每个请求一次只生成一个 token，保持流水线满负荷要困难得多。

### 数据并行

数据并行是三种方式中最简单的想法。不拆分模型。在不同的 GPU 上运行模型的完整副本，给每个副本不同的请求处理。

GPU 1 持有完整的 70B 模型，处理用户 1-100。GPU 2 持有完整副本，处理用户 101-200。它们从来不需要互相通信。没有 all-reduce，没有激活值传递，没有流水线气泡。每个 GPU 完全独立。

代价是内存。一个 70B 模型在 fp16 下是 140GB。每个 GPU 仅持有模型权重就需要 140GB——非常昂贵。

## 调度问题

除了原始硬件效率之外，还有另一个问题：调度。

想象两个请求同时到达。

请求 A 有 200K token 的 prompt，需要一次巨大的 prefill 操作。

请求 B 是一个正常聊天请求，已经在 decode 中，只需要下一个 token。

如果它们共享同一个 GPU 池，请求 A 可能延迟请求 B——即使它们的工作负载完全不同。这就造成了**队头阻塞（head-of-line blocking）**。一个长 prompt 的繁重请求可以拖慢所有人的延迟敏感型 token 生成。

所以问题不仅仅是计算 vs 内存。也是不同类型流量之间的干扰。

这让分离的理由更充分了。

## 分离（Disaggregation）

现在我们知道了：张量并行、流水线并行和数据并行都有不同的权衡，但它们都做了同一个假设：

Prefill 和 decode 运行在同一个 GPU 上——这才是真正的问题。

那么，如果 prefill 和 decode 根本不运行在同一个 GPU 上呢？这就是分离。

不共享一个 GPU 池，而是创建两个独立的 GPU 池，每个池为一个工作负载优化：

- **Prefill GPU** 处理入站 prompt 并构建 KV cache，使用激进批处理和张量并行
- **Decode GPU** 专注于 token 生成，针对内存带宽和并发性调优

Prefill 完成后，prefill GPU 只需通过网络将 KV cache 发送到 decode GPU，由 decode GPU 接管生成。请求流现在变成：

> Prompt → Prefill 集群 → KV Cache 传输 → Decode 集群 → 输出 Token

现在每个集群可以使用对其工作负载实际最优的并行策略——不再需要妥协。

Splitwise 论文于 2023 年发表，到 2024 年，许多现代服务系统已经开始采用它。

### 代价：KV Cache 传输

分离不是免费的 :/

当 prefill 完成时，KV cache 需要通过网络从 prefill GPU 物理移动到 decode GPU。对于长 prompt 和大模型，KV cache 大小很容易达到数百 MB 甚至数 GB。

这种传输表现为 prefill 完成和第一个 token 流式输出之间的一个延迟间隙——这直接伤害 TTFT，也就是分离本应修复的问题。

那么怎么处理它？有几种方法：

- **将传输与计算重叠**：你不需要完整的 KV cache 就能开始解码。你可以在已到达的 KV cache 块上开始解码，同时其余部分仍在传输。到 decode 需要后面的块时，它们已经到达了。这隐藏了大部分传输延迟。
- **使用快速互联**：同一节点内 GPU 间的 NVLink 比走网络快得多。生产系统的设计保持 prefill 和 decode GPU 在物理上靠近，以最小化传输时间。
- **传输前压缩 KV cache**：在发送前将 KV cache 量化到 INT8，将传输大小减半，质量损失极小。一些系统自动执行此操作。

在实践中，传输成本是值得的——专门的 GPU 池带来的收益远远超过网络开销。但这需要真正的工程才能做好，这是一个活跃的研究领域。

### 如今的应用

分离从一篇研究论文到生产基础设施的速度快得惊人。Splitwise 论文于 2023 年发表，到 2024 年，许多现代服务系统要么正在采用，要么正在积极实验分离。

- SGLang 已有分离集成
- vLLM 正在添加分离支持
- 主要云厂商在内部构建自己的版本
- **Mooncake**，即 Kimi（中国最大的 LLM 部署之一）背后的服务系统，完全围绕分离式 prefill 和 decode 构建

它传播得如此之快的原因其实很直观。随着上下文窗口变长（4K → 32K → 128K → 1M token），prefill 的代价成比例增长。以前在 4K 上下文中只是轻微烦人的干扰问题，在 128K 上下文中变得完全无法承受。分离从一个好的优化几乎一夜之间变成了硬性要求。

## 要点

这就是分离这件事。一旦你看清楚了，它听起来几乎过于明显。Prefill 和 decode 不是同一个工作负载，从来都不是，但我们一直把它们强行放在同一个 GPU 上，然后疑惑为什么延迟不可预测。

解决方案是两个池、两个任务，各自为所需做优化。KV cache 传输成本是真实但可控的。仅调度隔离这一点就值了。

如果你在关注这个系列：KV cache 是关于内存的，推测解码是关于速度的，连续批处理是关于吞吐量的。而这篇是关于意识到架构本身一直都是瓶颈。
