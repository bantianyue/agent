# Codex 在设计上太强了——这里是真正有效的工作流

Codex + 一个秘密工具是我们机构一直在跑的设计工作流……输出完全在另一个层次上。

我已经说了好几个月了。如果你的应用看起来像 AI 拼凑品，那不是 Codex 的问题。那是工作流的问题。

大多数构建者陷在同一个 loop 里。他们用 AI 发布功能。后端能工作。逻辑是可靠的。但界面看起来像所有其他 vibe-coded 应用。用户在两秒内决定它是否看起来真实。

旧的修复方法是雇一个设计师。等几周 Figma 文件。然后看着你的开发者解读那些设计并重建到代码里。等到设计上线时，你已经失去了 momentum 而且烧掉了不该烧的钱。

整个过程现在变成可选的了。

为了这个演示，我建了 Haven，一个房产移动 App 概念。完整设计系统 20 分钟。每个屏幕设计在一小时内。通过 Codex 输出生产级代码，一行 CSS 没改。

这是我目前在跑的工作流。每个用 AI 发货的构建者都需要理解它怎么工作。Moonchild 在左边设计 UI，Codex 在右边把它变成可工作的移动 App。

**Moonchild 到底是什么**

秘密工具是 Moonchild。

我测试了过去一年发布的几乎所有 AI 设计工具。大多数在第一屏看起来不错，但在你建第二页的时候就崩了。一致性是问题。Moonchild 以一种其他工具没有的方式解决了它。

它是一个对话式设计画布，从你的设计系统开始。不是从空白 prompt。不是从截图。从你实际的系统。你有三种方式把系统带进去：
- 从 Figma、GitHub 或在线链接导入
- 向它的 Agentic Design System Builder 描述你的品牌
- 从 Dribbble 或其他地方拖入设计灵感

一旦系统进去，你不用旧的 prompt 方式来设计。你粘贴一个 PRD。它根据你的系统生成完整的屏幕。你从那开始做 art direction。

导出路径是关闭 loop 的关键。Moonchild 提供了一个 MCP 连接器，把你的设计作为结构化输出直接交给 Codex。不是给 Codex 解读的截图。是实际的设计数据。组件、token、布局，所有东西。

这就是工作流。现在来看怎么实际使用它。

**大多数构建者用 AI 做设计的问题**

大多数构建者是这样的。他们提示 Codex 建一个新功能。它发货了。看起来还行。然后他们提示下一个功能。看起来稍微不同。到第五个功能的时候，应用看起来像五个不同产品拼在一起。

这就是人们说 AI 设计很烂时的意思。模型不烂。工作流烂。Codex 不知道你的设计语言是什么。每个 prompt 都是全新的上下文。每个屏幕都是凭猜测。每个组件都偏离上一个。颜色在变。字体在变。一切都在累积。

修复很简单。给 Codex 一个系统去遵循。然后给它实际的设计去构建。大多数工作流给其中一个。Moonchild 加上 MCP 连接同时给了两者。

**第一步：先建设计系统，始终如此。**

这是每个人跳过的步骤。也是最重要的。

在我们的机构，我们在碰任何屏幕之前为每个客户建一个设计系统。这过去需要一个设计师两天。Moonchild 大约 20 分钟。

对于 Haven，有几种方式把设计系统弄进 Moonchild。你可以从 Figma、GitHub 或在线链接导入。你可以向 Agentic Design System Builder 描述品牌。对于 Haven，我做的是我通常从一个 vibe 开始时做的事：从 Dribbble 取设计灵感。我拖入了三张移动房产应用的截图——那些画面准确捕捉了我想要的感觉——加上一段关于 Haven 是什么、为谁而做的简短描述。工具提取了设计语言并从零构建系统。

让我惊讶的是所有东西都可以预览。大多数 AI 工具，包括我上一篇文章写的 Google Stitch 工作流，给你一个扁平的 token 文件。颜色。字体。可能间距。这个工具做的不一样。它构建一个完整的组件库和一个图库。每个 badge、按钮、卡片和 chip 都以它在应用中的实际样子渲染。点击任何组件，你会看到它的实时效果，带有测试变体和尺寸的控制面板。预览下方直接是该组件的 JavaScript、CSS、示例和依赖。准备好拉入代码库。

图库是真正改变你设计方式的部分。不是猜测"品牌 badge"或"评级 badge"在上下文中应该是什么样子，你一眼看到它们全部并列。For Sale。For Rent。Featured。整个家族一次看到。你在任何屏幕被生成之前就做出有依据的设计决策。

这是一个风格表和一个真正系统之间的区别。

**第二步：PRD 到 UI**

一旦你的设计系统进去了，你不用旧的 prompt 方式来设计。

你粘贴一个 PRD。一页的功能简述。一个产品 spec。一个目标陈述。你拥有什么都可以。

对于 Haven，我粘贴了一个结构化 PRD，涵盖产品介绍、目标、目标用户和角色、关键功能和 MVP 范围、完整的用户旅程、技术栈和设计方向。大约一页。具体到给工具真正的上下文。结构化到把正确的细节拉入正确的屏幕。

然后我做了一件大多数人跳过的事。我在生成任何屏幕之前和它 brainstorming。这就是大多数构建者错过的解锁。聊天模式持有完整的 PRD 上下文。不是倾倒 PRD 然后点生成，你首先讨论产品。确认屏幕列表。锁定导航模式。在任何像素存在之前预定义关键屏幕的层级。

对于 Haven 我问了：根据这个 PRD，你推荐我们先建哪些屏幕？房地产应用我有没有漏掉什么？主导航应该是 Home、Search、Favorites、Profile 的底部 tab 还是别的？在生成之前，请带我走一遍 Property Detail 屏幕的优先级顺序——照片先，然后是名称和价格，然后是规格，然后是设施，然后是经纪人。确认或改进。好的，现在请根据我的设计系统，用亮色和暗色两种模式生成这些屏幕。确保完成整个用户旅程并创建所有屏幕。

当我点击生成时，工具对我产品的了解已经超过了大多数设计师在第一次客户电话后能有的了解。

关于变体的快速说明。仔细读这个。一个很酷的功能：你可以创建屏幕的不同变体。你只需要明确告诉 Moonchild，它会处理。对于 Haven，我想要三种不同的美学方向，然后才确定一个。我用的 prompt：我希望你为 Haven 生成三个完整的移动 UI 设计概念。概念 1——温暖编辑风：奶油背景、宽裕间距、衬线标题、柔和阴影、杂志风格。概念 2——大胆现代：深色模式、高对比度、无衬线、锋利边角、照片主导。概念 3——极简高级：中性调色板、精致字体、大量呼吸空间、微妙微交互。把已连接的设计系统作为基础，但让每个概念把系统拉向不同的美学方向。生成完整屏幕，不是片段。你会怎么做？

注意最后的"你会怎么做？"。这是让流程生效的小细节。不是盲目地生成所有东西，工具返回了一个计划。它说它会先为每个概念建 5 个代表屏幕。我审视三个方向。一旦我挑中一个，它就朝那个方向设计整个应用。这完全改变了成本方程。你付的不是三个完整构建。你付的是三个快速样本，然后一个在你实际选的方向上的完整构建。

我选了温暖编辑风。完整的屏幕集回来了。Moonchild 一次生成了完整的 Haven UI，包括亮色和深色模式版本跨越每个屏幕。

真正让我惊讶的部分是：Moonchild 生成的每个屏幕在预览模式下都是可交互的。不是静态模型。真正可交互。你可以点击输入字段。点击按钮。像真实用户一样在屏幕间移动，在有任何代码被写出来之前测试原型。对于 Haven，我在预览中发现了两个流程问题，否则会发给 Codex。Property Detail 屏幕没有明显的返回浏览的方式。联系经纪人的 CTA 太靠下了。如果我没有点击检查就直接交给 Codex，两个都会发过去。

这就是解锁。你在任何代码被写出来之前测试产品。

最后的优化 loop。当屏幕需要修改时，你不是从头重新 prompt。你选中屏幕，告诉工具要改什么，它在原地生成 v2。想要更电影化的 hero？v2。想把卡片布局换成列表视图？v3。变体叠加让你可以比较并选最强的。

对于 Haven 的 Property Detail 屏幕，我经历了三次迭代。v1 经纪人区块太高。v2 修复了层级但图片库感觉小。v3 两者都好了。大约 8 分钟的来回。

你不是从空白开始策展。你在优化一个真实的方向。这部分感觉最接近和高级设计师一起工作。

**第三步：通过 MCP 把 Moonchild 连接到 Codex**

这就是工作流变得超强的地方。

设计系统给 Codex 规则。MCP 连接给 Codex 更强大的东西：直接访问实际的设计输出。不是截图。不是设计的文字描述。Codex 可以读取和重建的结构化设计数据。

设置方式：进入 Moonchild Settings → MCP → 复制安装命令 → 打开 Codex CLI → 粘贴命令 → 用 Moonchild 账户认证 → 完成。

一旦连接，Codex 能看到你的设计系统和具体的 Haven 屏幕。

**第四步：Codex 输出代码**

现在你像这样提示 Codex：

"使用 Moonchild MCP 获取 Haven 房地产移动 App 项目，告诉我里面有哪些屏幕。"

第一个 prompt 做两件事。它确认 MCP 已正确连接。它给 Codex 对你设计的每个屏幕的完整可见性。

一个快速提示：在让 Codex 做任何更重要的事之前，保持在计划模式。Codex 在执行前会展示它要做什么。每次读计划。确保它理解你想要哪些屏幕、什么顺序、什么精确度。这是 Codex 写你实际需要的东西和写它认为你需要的东西之间的区别。

一旦你审查了计划并把它指向你想要的屏幕，Codex 直接从 Moonchild 拉取设计数据，引用你的设计系统，然后写代码。它甚至添加 hover 状态、转场和边缘情况处理，因为它在用实际的设计数据工作，而不是从 prompt 猜测。

结果：生产级 UI 进入你的实际代码库，与设计精确匹配，几分钟内。

在我们的机构，我们过去在每个客户 MVP 上为设计预算两周。设计师建 Figma 文件。开发者解读它。解读会漂移。我们修正。循环。用这个工作流，整个 loop 压缩成一个下午的工作。相同的质量。更高的利润。少得多的上下文切换。

**什么时候用这个工作流**

这不适合每个团队。诚实地评估什么时候适合。

适合的情况：你作为 solo builder 或小团队发货 MVP；设计一致性在跨屏幕时一直出问题；你不想维护一个独立的 Figma→设计师→开发者管线；你已有设计系统并希望它在各处被强制执行。

跳过的情况：你已经规模化，有完整的内部设计团队和成熟的 Figma 库；你需要像素完美的交付用于营销站点或品牌工作，每个阴影都重要；你的代码库有高度自定义的 UI 模式需要 bespoke 设计工作。

对于 80% 发货 MVP 和 SaaS 产品的构建者，这个工作流覆盖你需要的所有东西。

**需要注意什么**

在你跑这个之前，几个诚实的提醒。

你的设计系统质量决定你的输出质量。垃圾进，垃圾出。一个只有三个颜色 token 的半完成 Figma 文件会产生平庸的屏幕。在系统上投入时间，它在每个屏幕上都回报。

PRD 质量很重要。一个模糊的 brief 给你通用的屏幕。一个带有用户流程、边缘情况和清晰层级的 PRD 给你有用的设计。这不是工具限制。这是思考限制。

你仍然需要品味。工具生产。你指导。挑选正确的变体并优化它就是你的判断力所在。如果你跳过策展步骤，你就回到了 vibe 设计。

一些组件需要手动调整。复杂的动画状态、自定义交互模式、任何在设计系统之外的东西都需要在 Codex 里手动过一遍。MCP 把你带到 90%。最后 10% 是你的品味。

**这实际上意味着什么**

这是我诚实的看法。

用 AI 构建产品的瓶颈从来不是代码。AI 可以写代码。瓶颈一直是设计层。一致性、品味、把一切绑在一起的系统。

那个瓶颈刚刚崩溃了。

在我们的机构，我们过去在每个客户 MVP 上为设计预算两周。雇设计师。等 Figma 文件。看着开发者解读。循环。过去需要两周的现在需要两天。相同的质量。更快的交付。更高的利润。

这就是有资金的团队已经用 Figma+设计师+开发者管线拥有的工作流。区别在于现在 solo builder、独立机构和小型工作室也能获得同样的杠杆，不需要人头数。

2026 年对那些先搞明白这个的机构和构建者来说将是不公平的。

TLDR：停止抱怨 Codex 导致 UI 烂。Codex 不是问题。你的工作流是。第一步：在 Moonchild 里建你的设计系统。第二步：粘贴 PRD，在生成前用聊天模式 brainstorm。第三步：通过 MCP 把 Moonchild 连接到 Codex。第四步：用 MCP 提示 Codex 建每个屏幕。你的设计系统质量决定你的输出质量。Figma→交付→开发者的循环死亡了。一个工作流现在同时运行设计和代码。
