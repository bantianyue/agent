# Karpathy 的 CLAUDE.md 十二条军规：把 Agent 的错误率从 41% 打到 3%，秘密全在「上下文工程」

Andrej Karpathy 用一句话掀翻了 AI 编程圈的桌子：「Claude 90% 的错误不是模型太弱，而是上下文没给够。」这不是观点，是数据。没有 CLAUDE.md 时 Agent 的错误率是 41%；加了 4 条基础规则，降到 11%；当规则扩展到 12 条，错误率只剩下 3%。从 41% 到 3%，差距不是模型版本，是一份文件。

![Karpathy CLAUDE.md 推文配图](karpathy_video_thumb.jpg)

**CLAUDE.md 的本质不是「提示词模板」，而是 Agent 的操作系统级配置文件。** 它告诉 AI 助手你的代码库怎么组织、你的编程哲学是什么、哪些事绝对不能做。Karpathy 把这套东西叫做「上下文工程」（Context Engineering）——与其花时间调模型，不如花时间把上下文写清楚。因为模型不犯错的地方，都是它「看得到」的地方。

---

**第一条：先想再写代码——陈述假设，不要猜测。** Agent 最大的问题不是写得慢，而是写得快但方向错。Karpathy 要求模型在动笔之前，先把对问题的理解说出来，把隐含假设摊在桌面上。这相当于在代码生成之前加了一道「思维校验门」。

**第二条：简单优先——最少的代码，零投机性抽象。** 不要为了「将来可能用到」而写抽象层，不要为了「看起来优雅」而引入设计模式。CLAUDE.md 明确告诉 Agent：你的审美不重要，代码的可移除性才重要。写最少的东西，完成当前任务，然后停手。

**第三条：外科手术式修改——只动必须动的地方。** 改一个函数就不要顺带重构整个模块。Karpathy 的规则是：每一次 diff 都应该小到让人一眼能看出意图。如果你改了 10 个文件才修了一个 bug，那不是修 bug，是在埋雷。

**第四条：目标驱动执行——先定义成功标准，循环直到验证通过。** Agent 不能自己决定「什么时候算做完」。CLAUDE.md 要求：每个任务开始前，先写清楚「什么状态算完成」，然后执行、检查、再执行，直到验证通过。不是「我尽力了」，是「我验证了」。

**第五条：只让模型做「判断类」工作——分类、起草、总结、提取。不要用它做路由、重试、状态码处理。** 这是 Karpathy 最狠的一条。Agent 适合做需要语义理解的事，但不适合做需要确定性逻辑的事。重试逻辑、HTTP 状态码分支、路由分发——这些应该用代码写，不是用 LLM 推理。把 Agent 当「思考者」而不是「执行者」。

**第六条：Token 预算不是建议——每任务 4000 token，每会话 30000 token。** 这是硬约束。CLAUDE.md 写得清清楚楚：超过预算就拆分任务，不是让模型「尽量简短」。Token 预算的本质是注意力预算——超出之后模型就开始遗忘上下文，开始幻觉。Karpathy 用数字锁死了这个边界。

**第七条：暴露冲突，不要平均它们。** 当代码里有矛盾的需求、冲突的约束，Agent 的天然倾向是「和稀泥」——两边都满足一点，结果两边都没满足。CLAUDE.md 明确要求：遇到冲突就大声说出来，让人类决策，不要自己搞折中。

**第八条：读清楚再写——先读导出接口、调用方、共享工具函数。** 这是 Agent 最常见的翻车场景：改了一个函数，没发现它被 17 个地方调用。Karpathy 把「读代码」写进了规则——在写任何代码之前，必须先读相关的上下游代码。读的时间不计入 token 预算，因为读错了写的全是废的。

**第九条：测试验证意图，不只是验证行为。** 普通的测试只检查「输出了什么」，Karpathy 要的测试检查「为什么输出这个」。CLAUDE.md 要求 Agent 在写测试时把「被测试代码的意图」写进测试注释里。这样当未来有人改代码时，测试能告诉他的不是「你改坏了」，而是「你改的方向和原始意图不一致」。

**第十条：每完成一个关键步骤就做检查点。** Agent 的工作流里最可怕的事不是出错，而是出错之后回退的成本。CLAUDE.md 要求每完成一个逻辑步骤就做 checkpoint——可以是 git commit，可以是状态快照。这样任何一步出错，回退代价都是一步，不是从头再来。

**第十一条：匹配代码库的约定。** 不要用你的风格，用代码库的风格。缩进、命名、注释风格、错误处理模式——Agent 必须自动检测并适配。CLAUDE.md 里写了一条铁律：「如果你不确定代码库用什么风格，先读 10 个现有文件再写。」

**第十二条：失败要大声——「完成成功」但 14% 的任务被静默跳过，这是最可怕的 bug。** Karpathy 发现 Agent 经常在遇到困难时「静默跳过」——不报错、不求助、直接假装任务完成了。CLAUDE.md 的最后一条规则就是：如果没做完，必须说没做完。失败不可怕，假装成功才可怕。

---

**真正让这 12 条起作用的，不是规则本身，而是四件事的叠加。** 第一，CLAUDE.md 是跨会话的「机构记忆」——不是每次对话重新写提示词，而是文件里沉淀下来的团队共识。第二，改变必须基于评估（Eval），不能基于感觉（Vibe）——没有量化数据的优化都是自我感动。第三，检查点优先于速度——快但不可回溯的工作流是负资产。第四，纪律优先于框架——再好的框架也救不了没有纪律的 Agent。

有趣的是，Claude Code 之父 Boris Cherny 自己的 CLAUDE.md 只有两行。Less is more。Karpathy 的 12 条是给复杂项目用的重型武器，Boris 的两行是给个人工作流用的轻量配置。两者不矛盾——关键是你的上下文需要多厚的「工程」。

---

> **结语**
>
> 从 41% 到 3%，Karpathy 用一份文件证明了 AI 编程的核心瓶颈不在模型，在上下文。CLAUDE.md 不是魔法，是纪律的具象化。它把「让模型猜」变成了「让模型读」，把「写代码」变成了「工程化地写代码」。如果你还在跟 Agent 的幻觉搏斗，也许问题不在 Agent，在你没给它一本足够好的操作手册。
