# Claude Tag + Agent Identity — 完整翻译

## 第一部分：Introducing Claude Tag

**Introducing Claude Tag**
2026年6月23日

Claude Tag 是团队使用 Claude 的一种新方式。

我们从 Slack 开始，Claude 可以作为一个团队成员加入 Slack。授权 Claude 访问选定的频道，并将其连接到你选择的工具、数据——甚至代码库。然后，频道中的任何人都可以 @Claude 并委派任务给它，同时专注于其他工作。Claude 通过记住其所在频道中的相关信息来构建上下文，并且可以规划未来要完成的任务。

我们认为 Claude Tag 是 Claude Code 演进的开端：它让模型更加主动，并且与整个团队配合得更好。@Claude 现在是 Anthropic 内部完成工作的主要方式之一。今天，我们产品团队 65% 的代码是由我们的内部版 Claude Tag 创建的。同样的模式现在正在工程之外广泛传播——我们在用 Claude 追踪产品指标和数据、处理支持工单，甚至帮助找出棘手 Bug 的根本原因。

我们在 Slack 上推出 Claude Tag，因为它是团队与 AI 之间协作的自然场所，也是 Anthropic 大部分日常工作已经发生的地方。它今天以 Beta 版形式提供给 Claude Enterprise 和 Team 客户。我们的目标是将其扩展到更多可用的地方。

**与 @Claude 协作**

如果你用过 Claude Code 或 Cowork，Claude Tag 会让你感到熟悉。用简单的语言 @Claude 提出请求，它会将任务分解为多个阶段，然后依次完成。完成后，它会在 Slack 线程中回复。

@Claude 是多人的。在给定的 Slack 频道内，只有一个 Claude 与所有人互动。任何人都可以看到它在做什么，并且可以从上一个人离开的地方继续对话。

@Claude 会随时间学习。随着 Claude 关注其频道，它会构建更多关于工作的上下文。用户不需要一遍又一遍地向它解释事情。Claude 甚至可以自动从其他 Slack 频道和数据源学习（如果被授予权限的话，它不会从私密频道报告）。

@Claude 会主动行动。如果启用了"环境"行为，Claude 会主动让你了解它认为你可能需要知道的事情。它会从它所在的频道和连接的工具中标记相关信息，并跟进那些已经沉寂但尚未解决的线程或任务。

@Claude 异步工作。给 Claude 设定一个任务，你可以专注于其他优先事项。它还可以为自己安排任务，在数小时或数天内自主推进项目。

你也可以给 Claude 发送直接消息：它会使用你设置的个人工具和连接器私下回复。

**开始使用**

系统管理员指定模型应该访问哪些工具和信息，在哪些频道中。可以把它想象成为不同用途创建独立的 Claude 身份：所有内容，包括它的记忆，都将限定在管理员定义的频道范围内。例如，为销售工作设置的模型不会将其记忆传递给为工程设置的模型。

一旦权限设置完成，每个人都可以立即开始 @Claude。管理员可以设置 Token 消费限制，并且可以查看 @Claude 所做的一切的日志。

如果你是 Claude Enterprise 或 Team 客户，从今天开始你就可以使用 Beta 版。要开始使用，只需四步：将 Claude Tag 与你的 Slack 工作区配对，授予 Claude 访问你的工具的权限，设置月度消费上限，在私密频道中测试。Claude Tag 取代了现有的 Slack 版 Claude 应用，管理员可以在 30 天内选择迁移。Claude Tag 运行在 Opus 4.8 上。

## 第二部分：Agent Identity Access Model

**Agent Identity in Claude Tag：面向自主、团队级 AI 的新访问模型**

2026年6月24日

为了让 AI Agent 在人类-Agent 团队中发挥最佳作用，它需要访问人类拥有的相同工具、文档和上下文。

在"单人"AI 体验中（一个人与一个助手聊天），这很简单：你连接自己的账户，Agent 以你的身份行动。但在"多人"AI 体验中（如 Claude Tag），Claude 同时坐在一个共享频道中与多人互动，它使用的是属于工作区的工具和上下文，而非任何个人。

为了让多人体验正常工作，Claude 需要自己的工具账户，由管理员设置并与工作区绑定。我们称这种访问模型为 **Agent Identity（Agent 身份）**。

**为什么"以用户身份执行"行不通**

当你把 AI 当作个人助手使用时，你可以连接 Google Drive、GitHub 和日历等平台，让模型使用你的访问权限进行读写。这种模式在 Claude Tag 中行不通，原因有二：

Agent 自主性在增长。AI Agent 能自主完成的任务时长大约每四个月翻一番。Agent 现在可以为自己安排未来的任务，并在请求者下线很久后响应事件。Agent 在很大程度上自主工作。

多人团队。Claude Tag 将 Claude 放在团队已经在协作的共享空间中——例如，三个工程师和一个 PM 一起调试的频道。但当不止一个人在指挥时，谁的权限适用？没有单一的"正确人选"。这给了管理员定义 Agent 在 Slack 中能做什么的能力，独立于涉及的人类，并且可以清晰追踪 Slack 中发生的一切。

**Claude 以自身身份行动**

在启用了 Claude Tag 的频道中，Claude 不以单个用户的身份行动。它在它接触的每个系统中都有自己的账户：它在 Slack 中以 Claude 应用的身份发帖，以 Claude GitHub App 的身份打开 PR，以管理员配置的服务账户身份查询数据仓库。

因为没有个人用户凭证参与其中，共享频道永远不会成为访问某人私人文档的后门。

**权限继承**

在 Agent Identity 模型中，管理员在工作区级别定义一个身份——Claude 在任何地方都拥有的基础连接和技能集——每个频道默认继承它。然后，在需要的地方，他们可以在频道级别覆盖它，例如授予工程频道访问 GitHub 和数据仓库的权限，或将 CRM 连接限制在单个私密频道内。

除了凭证，管理员还定义：
- 仓库访问：Claude 可以读写哪些仓库
- 连接器：Claude 用来完成工作的工具和 API 密钥
- 技能和插件：Claude 动态加载的指令、脚本和资源文件夹
- 常设指令：每个频道的自定义指令和上下文

由于这种模型围绕不同的 Claude 身份运作，撤销身份就会终止 Claude 在该身份使用过的所有地方的访问。这比审计数十个用户账户中的单个 Agent 操作要容易得多。

**Agent Identity 模型的工作原理**

Agent Identity 将问题从"这个用户能做什么？"转变为"这个 Agent 在这个隔离区能做什么？"这与基于用户的访问控制列表不同：这意味着没有直接仓库访问权限的频道成员，如果频道的 profile 授予了 Claude 该权限，就可以让 Claude 读取该仓库。

Claude Tag 为每个私密频道创建一个独立的身份；公共频道共享一个工作区级别的身份。Claude 在法律频道的身份无法访问未授权的代码，它在工程频道的身份无法读取未授权的法律文档。记忆和访问都遵循这些边界：Claude 在私密频道中学到的东西永远不会出现在更广泛的工作区中。

身份属于频道，所以频道中的任何人都可以默认 @Claude。管理员可以将每个频道的 profile 限定为最低权限成员。在企业版上，基于角色的访问控制让管理员可以进一步决定哪些成员可以调用 Claude。

**直接消息**

在 Claude Tag 中，直接消息的工作方式与共享频道不同。DM 运行在用户个人的 claude.ai 账户上——使用他们自己的连接器、凭证和身份。这使得 DM 成为处理那些绝不应存在于频道中的任务和工具的正确场所，比如邮件草稿或只有你有许可证的软件。

**安全与审计**

当管理员向频道的 profile 添加连接时，凭证被独立存储并映射到该频道的身份，然后在请求时在网络边界注入。发往管理员未允许的主机的出站流量被直接阻止。在审计方面，使用 Agent 凭证进行的每个例程、记忆写入和网络调用都被记录。由于 Claude 以自有服务账户行动，这些操作也会出现在每个连接系统的日志中。

**下一步**

Agent Identity 是 Claude Tag 访问模型的基础。未来，Anthropic 计划增加即时凭证授权（用户可以在当下批准单个敏感操作，而不永久扩大 Agent 的范围），以及面向更复杂权限结构的身份感知覆盖层。

从单人 AI 到多人 AI 的转变，使得像 Claude Tag 这样的产品可以实现长期运行、基于团队的工作。Agent Identity 确保 Claude 对工具的访问足够广泛以发挥作用，但又足够限定以在企业规模下保持安全。
