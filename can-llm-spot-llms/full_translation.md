# 三模型问题：LLM 能认出其他 LLM 吗？

## 引言

这个博客的想法源于与 Joey 的讨论——我的上一篇博客是否有 Claude 编辑的痕迹，以及区分 Claude 和 GPT 的"气味"特征。作者也一直在思考 AI 写作问题，尤其是 X 上流行的长文，发现非常难以阅读。选择这个问题的动机：第一，这是个有趣的问题；第二，想找出我们擅长检测和识别的模型"气味"到底是什么，以及 RL 能否模拟这种能力。

这个看似简单的问题，最终占用了作者大约两周时间，把作者和同事带入了 RL 和相关后训练技术的优缺点深渊。

TL;DR：作者试图用 RL 训练 Qwen3.5-9B 仅凭文字区分 Claude、ChatGPT 和 Gemini。RL 不断停滞，SFT 达到了惊人的样本效率和准确率，一个线性分类器同时击败了它们俩。博客剩下的内容就是尸检报告：为什么 RL 一直失败，SNR 数学分析早就预言了这种结果，以及我们尝试的修复方案（OPCD/RLSD 风格的在策略自蒸馏）大多只是确认了诊断。

## 数据生成和前提假设

初始实验很简单：用 Opus4.8 和 GPT5.5 生成一批各种主题的博客，让 Qwen-3.5-9B 做 RL。考虑到数据生成中标签的可验证性，这看起来是个简单易行的任务。

初始数据集包含 126 篇博客，分为 ChatGPT 和 Claude 两组，覆盖 9 个类别：认知科学-神经科学、网络安全-基础设施、深度学习-AI、经济学-系统、历史、文学-文化理论、哲学、政治、STEM。

在 126 篇原始子集上做了初始 pass@4 分析，RL 有很大提升空间——pass@1 接近随机（50%），pass@4 约 94%。过滤掉全对和全错的组后做 RL。训练时保留 pass@4 最低的 3 个类别作为评估子集，其余用于训练。

| 指标 | 值 |
| pass@1 | 56.0% |
| pass@2 | 77.5% |
| pass@4 | 94.4% |
| 解析失败 | 0% |
| 截断 | 0% |

这个语料库后来发现太小了，无法提供有效信号。大多数 RL 运行在第 4 步就达到峰值，然后严重坍缩到两个 provider 之一。主要原因是整个语料库在我们的配置下只够 7 步 RL 就覆盖完 1 个 epoch。

第一次运行的观察结论是：需要大规模扩展数据生成，使分布具备多样化——即对模型有难度、能避免类别坍缩、有 RL 可学习的有效方差。

后续数据生成沿三个轴同时扩展：provider 标签、类别覆盖、每个类别的话题数。这样可以从磁盘恢复生成，只填充缺失的（类别×话题×模型×长度）组合。

Provider 轴从 4 个生成模型开始，最终坍缩为实验中使用的 3 路标签空间：Opus 4.8 → CLAUDE，GPT 5.5 → CHATGPT，两个 Gemini 变体（Gemini-3.1-pro-preview + Gemini-3.5-flash）合并为单一 GEMINI 标签（合并是在 RL 运行显示模型大多数时候只预测 gemini 后做的）。每个样本还生成了三种长度：short（约 2500 词/4k token，适合 16k RL 窗口和对比对）、long（约 5500 词/9k token，用于单文本 prompt）、epic（8-12k 词，有意超出窗口，不用于训练）。

类别轴同样重要。最初的 9 个类别偏技术和分析性，在二元实验中没问题，但 Gemini 加入后，只在技术语体上训练过的模型无法泛化到更具论证性和叙述性的验证集。因此从 9 类扩展到 21 类，增加了 12 个人文和社会科学类别，与保留的评估集在语体上匹配：社会学-人类学、法律-法理学、艺术-建筑-设计、宗教-神学、语言学-语言、教育学-教学法、心理学-行为、媒体-新闻-修辞、城市-环境-地理、商业-战略-管理、医学-公共卫生、音乐-表演艺术。

最后，每个类别追加了第二轮话题，使语料库达到约 630 个话题，每类约 30 个话题。最后一次扩展主要是抗记忆化措施：增加类内多样性，更好地测试模型是否真正学到了写作风格而非话题模板。

## 语料库按三个轴扩展

每个阶段增长一个维度——先 provider 标签，再类别，再话题轮次——直到数据足够多样化和平衡以运行 RL。构建过程是阶梯式而非跳跃式：将 provider 标签坍缩为清晰的 3 路空间，语体匹配类别（9→21），然后将话题轮次翻倍到约 630 个。

## 奖励塑造与类别坍缩

RL 运行有很多活动部件，问题会无声地恶化。奖励曲线可以在模型悄然消亡的同时继续攀升。以下是每次运行后最终检查的一组诊断探针——调试和检查运行的好指南：

1. 死亡组：GRPO 组中所有补全都得到相同标签的比例。全错组产生零优势，意味着梯度为零。如果某个类别的死亡组超过约 30%，模型即将完全放弃该类别。
2. 每类奖励和召回率：聚合奖励曲线是骗子。模型在平衡批次上 8/12 得分时报告奖励 0.67，但某个类别召回率为 0。必须按 provider 标签拆分查看。
3. 截断率：达到最大 token 上限的补全比例。高截断意味着模型在漫谈而非决策。
4. 验证曲线（val + val_ood）：val 是分布内类别，val_ood 是保留类别。观察两者差距可判断模型是在学习风格还是记忆。

### 坍缩循环

最大且最反复出现的失败模式是类别坍缩。一旦某个类别进入循环，GRPO 的零优势过滤器会使其缺乏梯度——这是一个优化器永远无法逃脱的稳态，即使整体奖励不断攀升。危险之处在于奖励持续上升：在平衡的 3 类批次中，如果模型搞定了其他两个类别，聚合奖励爬到约 0.67，而一个类别已经完全死亡。

另一个问题是，尝试修复一个类别的平衡会导致模型在三类间轮番出现差召回率。模型就是想坍缩到最容易拿到高奖励的模式。

### 课程学习和平衡尝试

第一个直觉是课程学习：按难度排序样本，先喂简单的后喂难的，使模型始终处于能力边缘。这没什么帮助，因为模型在大组大小下经常全对，等课程切换阶段时已经模式坍缩了。

接下来尝试了严格的类平衡批次构建：每批中每个 provider 样本数相等。这延迟了坍缩但未能阻止。根本问题是平衡输入不等于平衡梯度。

### 修复方案：对比对和三联体

真正的结构性修复来自改变每组的提问方式而非过滤或平衡。

对比对：给模型两篇来自不同 provider 的博客，问"哪篇是 [PROVIDER]？"这使得坍缩在结构上不可能，因为任何恒定标签策略在对比对中至少 50% 错误，总能生成混合组，梯度始终流动。

这有效了。首次得到了所有类别在训练中保持非零召回率的运行。但遇到了新问题：边际漂移。模型在对比对中能准确区分两个 provider，但在单文本评估时仍有类别坍缩。

熵奖励：对抗漂移先验，添加了熵奖励鼓励输出分布在三个标签间分散。这帮助了早期训练（最佳早期验证分数 0.466 来自熵奖励运行），但持续熵奖励使模型在需要低熵继续学习的阶段保持高熵。

熵衰减：自然的后续方案。早期高熵奖励保持所有类别存活，到第 30 步衰减到接近零让模型专注。这稳定了后期爆发。

三联体：最终的进化。不再用对比对，给模型三篇博客（每个 provider 一篇），让模型标注全部三个。这同时暴露所有边界，使边际更难被利用。奖励是部分正确分数（0、1/3、2/3 或 1），即使部分正确的组也有梯度流动。最佳平衡运行（三个类别同时保持召回率 > 0.30）来自三联体+熵衰减组合。

## 算法设计空间

获得三个类别都存活、稳定的约 0.40 检查点后，自然的问题是：天花板是算法本身吗？2024-26 年文献有很多 GRPO 改进版：DAPO、CISPO、DPPO、MaxRL、Dr. GRPO、ScaleRL。其中几个看起来完全为我们的失败模式量身定制。因此运行了一个干净的双臂消融实验，每个都是对最佳熵配方的一个变量改变。

prime-rl 的默认损失已经是 DPPO（概率差异上的全变分信任区域），其优势已经是 Dr. GRPO（均值居中，无标准差归一化）。所以实际要试的有两样：CISPO 和 MaxRL。

MaxRL 和 CISPO 都表现不佳，原因比数字更有用：

| 方法 | 最佳验证 | 发生了什么 |
| 熵（现有方案） | 0.398 | 截断 38%→2.5%，三个类别都存活 |
| MaxRL | 0.311 | 截断卡在约 37%，验证上限 |
| CISPO | 0.291→0.241 | 坍缩到 ChatGPT，截断上升 |

MaxRL 的原理是较难的 prompt（只要可解）应获得更多奖励，以保持 pass@k 并提升 pass@1。这解释了为什么 MaxRL 将学习信号集中在困难 prompt 上，但代价是模型从未学会简洁性，截断在 37% 徘徊。

CISPO 的消融本可以避免——大部分训练实际上是在策略的，重要性采样裁剪从未触发，因为它退化为普通默认损失，复制了原始的边际坍缩。

结论：对于在策略、易坍缩的分类 RL，现代信任区域变体几乎是空操作（它们只在离策略时与默认不同，而我们在策略）。优势重塑改变了行为但方向不对。真正起作用的杠杆是未约束类别边际上的抗坍缩压力，即已经在做的熵奖励。这是一个有用的负结果：排除了大量充满花哨估计器的消融空间。

## 保护可交付成果

整个前提是 RL 应该让模型在标签内阐述每个 provider 的风格特征。但奖励只检查标签。可预见地，到约第 24 步，约 11% 的正确 rollout 有空的标签。模型在优化我们测量的东西，默默杀死了我们在乎的东西——这是奖励破解。

朴素修复是加法格式奖励，但这是最容易破解的表面。于是用了乘法理由门控：只有答案正确且理由标签包含 ≥12 个不同字母单词时，rollout 才得 1.0 分。这按预期工作，理由通过诊断达到 1.00 并保持。最佳理由门控验证峰值 0.4005（宏召回率 0.40，接近均匀边际，GEMINI 召回率 0.455——史上最高）。

最佳 RL 配方（三联体+熵衰减+理由门控）产出的模型风格指纹很有意思：

- CLAUDE：散文式、论证性、哲学框架、 hedging、第一人称
- CHATGPT：实例演示、编号小节、大量 LaTeX/格式、教学支架
- GEMINI：ASCII 图、数学符号、自信的叙事语体、宏大隐喻

这些仅来自二元正确/错误信号。但数字 0.40 在三分类任务上只比瞎猜的 0.33 好一点。我们耗尽了奖励塑造、数据扩展、算法变体和损失修改。0.40 就是这个任务的天花板吗？

## 可分离性诊断

在花费更多生成预算或尝试另一个 RL 技巧之前，我们问了一个应该一开始就问的问题：这些数据到底有多可分离？

在 RL 模型学习的同一个训练集上训练了普通 TF-IDF（词 1-2grams）逻辑回归，并在同一个验证集上评估。训练集和验证集没有共享任何（类别、话题）对，所以这是诚实的未见话题泛化。还跑了一个去格式文本版本以确认没有作弊。

| 分类器 | 看到的内容 | 验证准确率 |
| 逻辑回归，词 1-2grams | 全文 | 100% |
| 逻辑回归，字符 3-5grams | 全文 | 100% |
| 逻辑回归，仅前 1500 字符 | 全文 | 100% |
| 逻辑回归，仅功能词和标点 | 零内容 | 94% |
| RL 训练后 Qwen3.5-9B（最佳检查点） | 全文 | 40% |

一个线性词袋模型在从未见过的话题上完美分离了三个 provider。仅用功能词（the、of、may、we）和标点频率，去掉所有内容词后仍达 94%。信号不微妙、不依赖话题、不隐藏在文本深处——就在前 1500 字符里，在风格里。

线性模型依赖的特征与 RL 模型摸索的特征完全一致：

- CHATGPT：may、may be、for example、such as、not only、depends on（hedging 和枚举）
- CLAUDE：you、genuinely、precisely、honest、worth、exactly（对话式和真诚）
- GEMINI：furthermore、we must、profound、mathematically、paradigm（宏大和正式）

在最终数据扩展后（5258 篇博客，21 个类别，630 个话题）2 倍规模上确认了结果：100% 验证准确率，分布在完全不重叠的类别上也是 100%，仅功能词仍达 94%。Provider 语体不是话题伪影也不是语体伪影，它是一个稳定的低层风格签名，线性模型轻而易举就能读取。

约 0.40 的天花板从来不是数据问题。类别是微不足道的完美可分离的。RL 训练的 9B 模型把 60 个百分点的信号留在了桌面上。瓶颈是利用率，不是信息量。从零开始的标量奖励 RL，在 thinking 关闭的情况下，对于有监督学习轻松搞定的判别任务来说，就是一个极弱的教师。

### SFT 解决了它

#### 速查表探针（零训练）

在任何微调之前，尝试了最便宜的干预：将经验得出的每 provider 风格特征写入系统 prompt 作为"速查表"，评估基础模型。

| 条件 | val | val_ood |
| 基础模型 + 速查表 | 0.674 | 0.592 |

仅仅把特征注入 prompt 就带来了 +0.22 的提升，零训练。模型显然能用这些特征，只是没有提示时不会自动使用。

#### Gold-conditioned 教师 SFT

然后运行了正式的 SFT。数据生成：对训练集中每篇博客（2682 篇，每类 894 篇），给基础模型博客内容+速查表+告知正确答案的分析师注记，让模型写一段简短的、基于事实的推理理由说明哪些风格特征指向该 provider，然后是正确答案标签。这是 gold-conditioned 教师：它知道答案并写理由来证明。这样得到三个 provider 都完美类平衡的训练数据。

然后在这些教师生成的示例上训练模型（使用普通 prompt）。配置：Qwen3.5-9B，thinking OFF，正文截断到 3000 token（序列长 4096），lr 1e-5 cosine，270 步。

| 检查点 | val | val_ood | ChatGPT 召回率 |
| base | 0.454 | 0.410 | 0.19 |
| SFT 第 90 步 | 0.906 | 0.909 | 0.98 |
| SFT 第 180 步 | 1.000 | 1.000 | 1.00 |
| SFT 第 270 步 | 1.000 | 1.000 | 1.00 |

分布内验证和交叉语体 OOD 集都达到完美准确率，所有类别平衡。885 个评估样本零错误。历史上无望的 ChatGPT 类（base 召回率 0.19）达到 1.00。

遗忘检查（通用文本困惑度+能力探针）显示 SFT 几乎零成本：困惑度仅 +3.2%，所有算术/推理/代码探针完整。

#### STaR 自蒸馏与仅答案控制

一个有趣的后续问题是：模型能否从自己的推理中学习，而无需在生成阶段被告知 gold 答案？

运行了 STaR 流程。对每篇训练博客，基础模型生成推理（普通 prompt，无速查表，k=3，temp 0.7），只保留多数投票匹配 gold 标签的。对于仍然错误的博客，给速查表（非 gold 答案）再试。最终得到 1913/2682 个接受的理由（71%），类平衡到每类 357=1071 行 SFT 数据。

| 方法 | val | val_ood |
| STaR 自蒸馏（推理） | 0.932 | 0.953 |
| 仅答案（相同 1071 行门控数据） | 1.000 | 1.000 |

STaR 推理运行达到 0.93，令人印象深刻。但决定性控制实验：从相同 1071 个示例中剥离推理，只监督答案 token，重新训练——达到了 1.000/1.000。在相同数据上，仅答案 SFT 击败了推理 SFT。

这是整个项目最令人惊讶的结果。自生成推理不仅不是监督目标所必需的，反而是有害的。模仿模型自身（有时是虚假的）推理注入了噪声，稀释了标签梯度。损害集中在 ChatGPT（推理运行召回率 0.80-0.89，仅答案召回率 1.00），该类别的自生成推理最弱。仅 1071 个自门控行上的仅答案 SFT 匹配了 gold-conditioned 教师 SFT（2892 个黄金推理行的 1.000）。一个小型、干净、类平衡的标签子集就足够了，且对 OOD 鲁棒。

## RL 的最后一次公平尝试

在关闭 RL 篇章之前，使用两种互补设置给了 RL 最后一次公平机会。

首先，完全移除推理通道，训练 GRPO 仅输出答案。相同 base init，相同 3 路任务，但补全强制只有答案标签，没有理由标签。这直接测试了长推理轨迹是否稀释了稀疏奖励信号。

这次运行确实突破了旧的大约 0.40 无速查表平台，峰值约 0.65/0.66。但作为稳定的 3 路方案仍然失败：策略收敛到一个退化的二类边界（ChatGPT vs Gemini 强，CLAUDE 被吸收到 CHATGPT），训练最终停止在零可训练批次。

第二次运行了"帮助 RL 成功"的设置：速查表引导的 GRPO，将训练得出的风格提示注入系统 prompt，给基础模型一个非坍缩的起始地板。

操作上看起来很强。理由门控验证奖励从 0.443 上升到 0.662。关键问题是内化：

| 条件 | val | val_ood |
| 带速查表（训练条件） | 0.659 | 0.650 |
| 无速查表（内化检查） | 0.348 | 0.378 |

最后一对 RL 运行讲述了一致的故事。仅答案 RL 可以提取真实信号但坍缩到退化策略。速查表引导的 RL 可以优化行为并保持训练健康，但一旦提示移除就无法内化判别器。

## 为什么 RL 可能不是这里的最佳工具

经验模式很清晰。有趣的问题是为什么。有一个信噪比论据可以解释我们看到的一切，也说明 SFT 总是会赢。

### 一个 token 承载全部奖励

单个 GRPO rollout 是一个长 token 轨迹，但只有最终的标签 token 被评分。奖励信号是末端的一个比特。关于"我们分类对了吗？"的每个梯度都必须通过这个高亮 token 回流。其他约 4095 个 token 不携带任何任务信号。

每组训练信号由组的混合程度决定。如果 q 是模型当前正确类的概率，v = q(1-q) 是每个 rollout 的方差，梯度信号尺度为 G·v，噪声为 √(G·v)，信噪比约 √(G·v)。组大小 G=12 时，即使在最佳 q=0.5，SNR 上限约 1.7。对于单个优化步来说这是极小的 SNR。

### 最有信息量 = 最不确定

每个 rollout 的奖励方差 v = q(1-q)，在 q=0.5 处达到峰值。信噪比 √(Gv) 最大处也是学习信号最大处。残酷的转折是：最大学习信号点（q=0.5）也是最大不确定点。即使在最佳情况下，12 的组大小，每步 SNR 仅为约 1.7。

### 熵交叉点

熵奖励 β 分布在数百个推理 token 上，淹没了那一个答案 token。奖励保持平坦直到 β 衰减到交叉点 β* ≈ 0.003 以下。

我们用来阻止类别坍缩的同一个熵奖励，在整个早期阶段恰恰是淹没学习信号的东西。奖励只有在 β 低于 β* 后才开始移动。

### 为什么 SFT 胜出：√L 更密集的监督

RL 监督每个示例一个 token；SFT 监督每个 token。在长度 L 的序列上，这相当于有效信号上的 √L 倍优势——大约一个数量级。相同数据，相同模型——差别在于梯度来自哪里。密集的逐 token 目标给 SFT 约 √L ≈ 26 倍于单个 RL 奖励的有效信号。

这个分析还给出了具体建议。将熵奖励掩码到答案 token，或从一开始保持 β ≤ β*。通过增加 G 而非烧更多步来提高 SNR。如果需要速查表地板和内化，在训练过程中按课程收回速查表，使权重内化特征的奖励梯度不再为零。这些都不会改变这个特定任务的底线，但会让 RL 表现得不那么病理。

诚实的总结是：这个任务的形状不适合 RL。判别信号密集、无内容、线性可分离。正确的工具是小标签集上的密集监督，仅答案 SFT 结果（1071 个自门控行上 1.000/1.000）就是这种实践的体现。没有冷启动的纯 RL 试图通过一个数学上不太可能获得信号的通道同时进行特征发现和分类器稳定。

RL 并非无用。它只是被用在了舒适区之外。在基础模型中已存在正确特征、缺失的是搜索/排序/校准的问题上，SNR 数学看起来非常不同，RL 会得到回报。

## 在 RL 自己的框架内修复它

对 RL 运行的分析给我们提供了思考和工作的方向：问题可以通过更密集的信用分配得到缓解，如果有一种在策略的密集分配后训练方法，理论上应该有效。

### 更密集的信用分配：OPCD 和 RLSD

问题是一个 token 承载数千个 token 的奖励。明显的修复是给每个 token 一点信号。自然的做法是在策略自蒸馏：让教师逐 token 评分学生的 rollout，使用教师/学生 logprob 差距作为逐 token 梯度。如果教师显著优于学生，差距在所有地方都是信息性的，不仅仅是答案 token。

使用同一策略同时作为学生和教师：学生用普通 prompt rollout，教师使用相同权重但将训练得出的速查表拼接到系统 prompt 中。教师阅读相同的博客，上下文中已有风格特征，所以它像一个知道该看什么的人一样评分 rollout。

两个变体：

- OPCD（在策略上下文蒸馏）：逐 token 损失就是分离的教师-学生 logprob 差距，视为策略梯度信号，信任区域基于差距的符号。损失中无验证器。
- RLSD（验证器锚定自蒸馏）：逐 token 权重为 (PT/PS)^sign(A)，PPO 裁剪学生比率。验证器提供方向，速查表教师提供逐 token 幅度。

预检查诊断确认教师确实优于学生后才烧算力。

| 方法 | base/第 0 步 | 峰值（第 12 步） | 最终（第 40 步） |
| OPCD | 0.222，截断 38% | 0.291，截断 25% | 约 0.00，截断 96% |
| RLSD | 0.222，截断 38% | 0.409，截断 9% | 0.00，截断 100% |

RLSD 在第 12 步是该项目产生的最佳推理通道数字。普通验证准确率从 0.222 跳到 0.409，截断从 38% 降至 9%。

然后两个运行都恶化了。原因是相同的：基于采样 token 的教师-学生差距是反向 KL 风味的目标函数，反向 KL 倾向于模式寻找。无约束时，两个运行都坍缩到一个永远在 token 预算内不终止的速查表风格推理模式，导致重度截断、零梯度信号。

RLSD-anchored 变体添加了逐步信任区域。正如理论预测的那样工作——信任区域将策略锚定到行为策略 μ，阻止了悬崖。但它也压平了峰值：验证在 0.18-0.25 区间内，与 BASE 第 0 步无异。

### 热启动：SFT 先，RL 后

看了 SFT 运行的惊人结果，我们想：如果先用 SFT 做冷启动再做 RL 会怎样？

配方：从 BASE 在 1071 个 STaR 自蒸馏轨迹上做 60 步 SFT，然后是用在其他运行中的标准 RL 配方。

| 步数 | 0 | 8 | 16 | 24 | 32 | 40 |
| val | 0.670 | 0.704 | 0.771 | 0.773 | 0.819 | 0.895 |
| 截断% | 0 | 0 | 0 | 0 | 0 | 0 |

验证准确率从 0.670 近乎单调地上升到 0.895，截断在每一步评估和训练中保持为零。

之前的每个推理通道运行同时被要求做两件事：学习判别器并学习如何产生一个有界的良好结构的推理轨迹。经过 60 步 SFT 预热后，第二个问题在第 0 步就已经解决了。RL 只需要做第一件事，验证器奖励变成了干净信号，因为每个 rollout 都干净地终止。模式寻找失败模式消失了，因为没有退化的不终止模式可寻。

冷启动 RL 达到 0.895，仍低于仅答案 SFT 的 1.000。结果的价值不在于它击败了 SFT（它没有），而在于它隔离了为什么其他每个推理通道 RL 运行都坍缩了：是冷启动的问题，不是通道本身的问题。在这个任务上，热启动的 RL 运行行为良好且富有成效；base init 的 RL 运行则不然。

## 模型实际学到了关于三个 provider 的什么

我们在相同的 414 篇验证博客上用普通 prompt 运行了所有训练过的有用检查点的每个模型，保留了推理，并阅读了它们。

### 每个健康模型收敛到的共享特征词汇

最引人注目的发现是：每个产生连贯推理的模型，无论训练配方如何，最终都描述了相同的三种风格签名。来自 gold-conditioned SFT 模型的逐字引用：

1. CLAUDE：使用诚恳副词如 genuinely、honestly、precisely，配合散文式措辞。声音温暖且论证驱动，使用第二人称 you。
2. CHATGPT：典型的 hedging 和枚举风格，频繁使用 may be、can also、for example 等短语。结构严重依赖显式列表、编号步骤和形式化定义。
3. GEMINI：大量使用 intensifiers 如 profound、fundamentally、massive，自信且陈述性的语气。结构依赖复杂的数学形式化和 ASCII 图。

这三段话基本上就是这个项目。Claude 是使用诚恳副词和第二人称的散文家。ChatGPT 是构建编号结构和实例的 hedging 者。Gemini 是追求 intensifiers、ASCII 图和密集 LaTeX 的宏大形式主义者。

### 训练配方塑造推理的"性格"

同一特征根据通道训练方式的不同，交付方式差异很大：

| 模型 | val | 中位推理长度 | 具体性 | 自信度 | 预测分布 |
| Gold-conditioned SFT | 1.000 | 73词 | 0.98 | 0.35 | 平衡 |
| STaR 自蒸馏 | 0.952 | 82词 | 0.98 | 0.57 | 平衡 |
| 冷启动 RL | 0.911 | 81词 | 1.00 | 0.36 | 平衡 |
| 冷启动 SFT | 0.696 | 85词 | 0.99 | 0.46 | 平衡 |
| 纯准确率 RL（峰值） | 0.384 | 91词 | 0.98 | 0.74 | CHATGPT 336/414 |
| 速查表 RL | 0.372 | 15词 | 0.33 | 0.07 | CLAUDE 342/414 |
| 熵衰减 RL | 0.290 | 23词 | 0.50 | 0.04 | GEMINI 346/414 |
| RLSD/OPCD（坍缩） | ≈0.00 | 2000+词 | 0.10 | 0.00 | 100% 截断 |

### 关于通道的结论

特征是可学习且可言语化的。每个 grounded 模型独立读取相同的三种风格签名，它们命名的内容与线性探针一一对应。没有隐藏表示问题。不太舒服的结论是：推理文本很大程度上是对一个决策的事后叙述，这个决策是基于底层的词汇和标点统计。gold-conditioned SFT 模型写出优美的段落描述诚恳副词和宏大形式主义，但底层仅功能词的逻辑回归已经用同一信号拿到 94% 了。

如果从头开始，三件事会改变：第一天就运行可分离性诊断；将工具匹配到任务形状；永远不要在基础模型上为不能限制的任务启动推理通道 RL。

## 无 AI 痕迹的永恒阳光

作者想研究这个的更奇怪原因之一是：弄清楚 LLM 为何能生成信息密集的文本，但一旦 AI 感触发作就完全读不下去。作者坚信写作中使用 AI，不会回到非 AI 时代。模型在生成"优秀"的文本方面异常出色，只是不"人类"，而这个差距正是阅读困难的原因。

Leo 的推文抓住了这些模型的实际情况：在编辑和润色方面超人类，但在原创思想或写作方面则不然。作者也发现自己在做同样的事——把原始想法扔给模型看看返回什么，十之七八模型比脑子里的表达更好。

仍然阅读人类写作有真实价值。没有什么能比得上你能感受到一个人真正沉浸其中的作品。整个项目中一直萦绕的一句话：如果你不把努力和关心放在写作上，别人为什么要花精力去读它。所以给想写作的人：用你自己粗糙的语言写出来，确保浓缩进文字的是你，而不是一个矩阵。

### B. 词袋 prompt 消融（E-LEX）

可分离性诊断告诉我们线性词袋模型通过读取判别性 token 获得 100%。自然而然的后续问题是：如果我们直接告诉 LLM 像逻辑回归一样行为，让普通 RL 做剩下的事情呢？E-LEX 实验就是那个消融。

40 步运行看起来是项目中最干净的结果。但 100 步重运行时结果没有复现。诚实的解读是：在这个规模和配方下，普通 GRPO 的逐运行方差主导了词袋 prompt 的效果。40 步的攀升是一个真实运行，但不是方法的属性，而是种子的属性。

两项经验：词袋 prompt 不会给未训练模型任何免费的准确率；未来的推理通道声明在这个任务上需要至少两三个种子才能称"攀升"为真实。
