**Sakana Fugu 是一族 AI 模型，不自己回答问题，而是读取你的问题，决定哪些强大的模型应该处理它、它们如何协作、如何组合答案。**

Fugu 是指挥家。指挥家不演奏任何乐器，他读乐谱，决定哪个乐手在什么时候以多大音量演奏，把众多演奏者变成一场交响乐。Fugu 对 AI 模型做同样的事。

Fugu 来自东京的 Sakana AI 实验室，其联合创始人 Llion Jones 正是《Attention Is All You Need》的共同作者。他们称 Fugu 为**编排器模型（Orchestrator Models）**——取一组强大模型，让它们像一个更智能的单一系统那样运作。

**为什么需要 Fugu**

今天有 GPT-5.5、Claude Opus 4.8、Gemini 3.1 Pro 等前沿模型，但**没有一个模型在所有任务上最强**。论文观察到明确分工：GPT 模型在数学以及规划和整合思路方面最强；Claude Opus 模型在软件工程和发现安全漏洞（调试）方面最强；Gemini 模型在直接实现已知算法以及化学和生物学等科学领域最强。

面对这种局面，有几种尝试过的方案：

**方法 1：用一个模型做所有事。** 最简单，但受限于该模型弱项。

**方法 2：手动搭建固定团队。** 写死"模型 X 写代码，模型 Y 审查"，但僵化，对每个问题用同一套方案。

**方法 3：简单路由器。** 训练一个小的路由器看完问题就发给一个模型。但仍是单步决策——一次选一个模型，不能中途改变主意，也不能让两个模型协作。

**Fugu 的方案是可学习的编排器：** 读取问题，为这个特定问题构建计划——哪些模型参与、它们如何通信、何时合并答案。计划动态变化。

**大局：** 你给 Fugu 一个问题，它从池中选正确的模型或模型团队，返回最终答案。从外部看就像跟一个单一模型对话。论文指出**复杂性不来自任何单个部分的难度——来自各部分的堆叠方式。**

**集体智能与扩展轴**

集体智能：一组有限的个体通过协作产生比任何成员单独更聪明的行为。一只蚂蚁简单但蚁群建复杂巢穴，一个人知道有限但好团队解决大问题。**Fugu 将每个前沿模型视为专家，把它们变成一个团队。**

**论文将编排视为一条新的扩展轴（scaling axis）。** 过去让 AI 更强的方式是训练更大更贵的模型。Fugu 提出另一条路：让已有模型高效协作。

这有几个现实优势：新模型发布时加入池子即可，不重新训练任何东西；可以移除模型、偏向某个供应商、或出于隐私政策封禁模型，同样无需重新训练。模型因访问限制、宕机或价格变化不可用时，Fugu 自动绕开。团队中即使一个成员离开，整个系统继续工作。

**两个 Fugu——Fugu 与 Fugu-Ultra**

论文发布两个版本。**核心取舍是质量与延迟（等答案的时间）。**

- **Fugu：** 对每个输入选一个最佳工人模型，速度约等于直接调用一个模型，适合日常编码和聊天。基于 **Trinity** 研究
- **Fugu-Ultra：** 为每个输入构建多模型协作计划，追求最高质量，面向最复杂任务。基于 **Conductor** 框架

论文将两个版本的关键差异以表格呈现：Fugu 目标平衡速度与质量，每输入用一个工人，快速；Fugu-Ultra 目标最大质量，每输入用多个工人的计划，较慢。Fugu 最适合日常编码和聊天，Fugu-Ultra 适合最难的多步问题。Trinity 和 Conductor 是 Sakana AI 的两篇早期研究论文。Fugu 基于 Trinity 构建，Fugu-Ultra 基于 Conductor。**两个变体共享同一个核心思想：编排，但实现的复杂度和延迟代价完全不同。**

**Fugu 如何选模型：轻量级选择头**

Fugu 使用一个普通语言模型作为骨干（大脑）。输入问题时，模型读取形成**隐藏状态 h**——一组捕获问题含义的数字。**这是整个决策链的核心信号。**

通常语言模型有一个 LM Head 将 h 转换为下一个词。Fugu 在旁边加了第二个小型 Head，称为**轻量级选择头（Lightweight Selection Head）**。它取 h 并输出 L 个 logits，每个对应池中一个工人模型，分数最高的被选中。

几个符号：h 是隐藏状态（捕获问题含义的数字列表），L 是池中工人数量，logits 是原始分数（每个模型一个，在转为概率之前）。

**关键技巧：Fugu 用 logits 而非生成文本来决策。** 它在早期位置计算隐藏状态，立即用选择头读分数、选出模型、发给工人。因为跳过了缓慢的逐词生成，决策非常快。这就是 Fugu 低延迟的原因。

这也是 Fugu 与早期模型 Trinity 的区别。Trinity 还给所选模型分配角色（思考者、工人、验证者），Fugu 去掉角色总是将所选模型用作工人。更少的选择意味着更小更快的决策。

为了廉价调整大脑，Fugu 不重新训练整个模型。把模型想象成数百万个微旋钮——Fugu 只调一小部分（通过 SVD/奇异值分解，来自 Transformer-squared 论文），其余固定。选择头本身也保持小巧简单。

**第一阶段训练：有监督微调（SFT）**

Fugu 必须学会哪个模型最适合哪类问题。第一阶段是有监督微调（SFT）。

团队收集大量已知正确答案的问题。对每个问题让每个工人模型跑多次并打分。奖励在 0-1 之间，1 完美解决。

假设池中 3 个模型（GPT、Opus、Gemini），给数学问题跑多次平均：GPT 0.9，Opus 0.6，Gemini 0.3。分数向量 s = (0.9, 0.6, 0.3)。

简单做法是直接说"GPT 最好"——但丢弃了有用信息，隐藏了 Opus 也相当不错的事实。

**论文改用 softmax 函数将分数转化为软目标分布。** Softmax 取一组分数，转化为概率——所有为正、加起来等于 1、更大分数保持更大。

公式：p(j) = exp(r_j / T) / Σ exp(r_j' / T)。p(j) 是模型 j 的目标概率，r_j 是平均奖励，T 是温度（控制分布尖锐程度），exp(x) 是 e^x（e≈2.718）。

用 T=1 一步步算：exp(0.9)=2.460, exp(0.6)=1.822, exp(0.3)=1.350。求和=5.632。然后 p(GPT)=2.460/5.632=0.437, p(Opus)=1.822/5.632=0.324, p(Gemini)=1.350/5.632=0.240。

软目标 (0.437, 0.324, 0.240) 的意思是 GPT 约 44%、Opus 约 32%、Gemini 约 24%——**没说"总是用 GPT"**。它保留了 Opus 也是强备份的事实。更小的 T 让分布更偏重最佳模型，更大的 T 让模型更平等。

Fugu 的预测分布与软目标通过 **KL 散度** 比较，衡量两个概率分布之间的差距。

训练损失 L_SFT = (1/N) Σ D_KL(p(⋅) ∥ π(⋅|q))，其中 π(⋅|q) 是 Fugu 对问题 q 的预测分布，p(⋅) 是软目标。对所有训练问题取平均。

假设 Fugu 当前预测 (0.5, 0.3, 0.2)，目标 (0.437, 0.324, 0.240)。KL 散度公式 D_KL = Σ p(j) ⋅ ln(p(j)/π(j))：

- GPT 项：0.437 * ln(0.437/0.5) = 0.437 * ln(0.874) = 0.437 * (-0.135) = -0.059
- Opus 项：0.324 * ln(0.324/0.3) = 0.324 * ln(1.080) = 0.324 * 0.077 = 0.025
- Gemini 项：0.240 * ln(0.240/0.2) = 0.240 * ln(1.200) = 0.240 * 0.182 = 0.044
- 求和 = -0.059 + 0.025 + 0.044 = 0.010

KL 散度约 0.010，因为 Fugu 的猜测已经接近目标。当两个分布完全匹配时 KL=0。这些小数字只是为了理解，实际训练使用大量问题和许多模型。

**第二阶段训练：进化策略（sep-CMA-ES）**

第一阶段用单步问题训练，能清晰给每个模型打分。但真实编码是多轮的：读仓库、编辑文件、运行工具、读错误再试。第二阶段在完整端到端任务上训练。

问题在于：端到端任务只到最后才知道是否完成，得到单一奖励（1 完成/0 未完成）。没有清晰的逐步信号让正常的梯度训练生效。

**论文使用进化策略 sep-CMA-ES——模仿自然进化：尝试大量微小变异，保留表现最好的，向它们移动。** 不需要梯度，只衡量每个变体的表现。

用微小例子说明。假设当前值只是单一数字 θ=0.0，步长 σ=1.0，尝试 4 个变异：

步骤 1：加微小随机扰动 z 产生候选：z1=0.5→θ1=0.5, z2=-0.3→θ2=-0.3, z3=1.0→θ3=1.0, z4=-0.8→θ4=-0.8。

步骤 2：在端到端任务上衡量适应度（完成任务比例）：θ1=0.8, θ2=0.5, θ3=0.9, θ4=0.2。

步骤 3：保留前 2 名优胜者：θ3(0.9, z3=1.0) 和 θ1(0.8, z1=0.5)。

步骤 4：将父代向优胜者移动，权重偏更好者（假设 0.7 和 0.3）：新 θ = 0.0 + 1.0 * (0.7*1.0 + 0.3*0.5) = 0.85。

父代从 0.0 移动到 0.85，朝完成了更多任务的区域。这个循环重复许多代，Fugu 慢慢学会在真实多轮工作中更好地选模型。

论文中的完整公式：θ_k = θ_t + σ_t D_t z_k 创建候选（z_k 从正态分布采样——扰动大多很小偶尔很大），θ_{t+1} = θ_t + σ_t D_t Σ w_j z_j 更新父代（w_j 是组合最佳候选的权重）。D_t 是对角矩阵让每个方向被不同量级探索。**核心想法和上面微小例子完全一致。** "sep"表示"可分离"——只保留对角线调优每个值而不追踪对值关联，更便宜且可处理大量参数。这与训练 Trinity 的进化方法相同。

**Fugu-Ultra 的 Conductor：写完整 Agent 工作流**

Fugu-Ultra 基于 Conductor 框架构建。Fugu 每步选一个模型，Fugu-Ultra 更进一步：**写出完整 Agent 工作流让多个模型在同一个问题上协作。**

每个工作流是一系列步骤，每步三个部分：**子任务**（通俗语言指令告诉工人做什么）、**工人 ID**（指定哪个模型做）、**访问列表**（指定能看到哪些先前结果）。Fugu-Ultra 不仅选模型，还为每个模型写指令、选谁做什么、连接谁能看到谁的工作。

因为 Fugu-Ultra 控制访问列表，它能构建多种协作形态：**简单链**（一个模型接一个工作）、**Best-of-N**（多个模型独立尝试同一件事后一个模型挑最佳）、**树**（多个并行工作后一个聚合器合并）。

树的例子：困难冷知识问题，Fugu-Ultra 让 Gemini 和 GPT 各自独立尝试，然后让第二个 Gemini 通过访问列表阅读两个尝试合并答案。两个叶子节点都有部分错误，但 Gemini 聚合器发现每部分的正确之处，产出完全正确答案。

**关键在于 Fugu-Ultra 能根据问题选不同的模型做聚合器。** 冷知识用 Gemini（强于冷门知识），数学用 GPT（强于数学）。旧系统被迫始终用同一个模型做最终组合——Fugu-Ultra 在每个问题上都能自适应。

**教 Fugu-Ultra 指挥：GRPO 强化学习**

用 GRPO（Group Relative Policy Optimization，来自 DeepSeekMath 论文）训练。

奖励规则：对每个工作流，格式错误不可读→r=0；运行并给出正确答案→r=1；运行但答案错误→r=0.5。Fugu-Ultra 既因写出有效计划也因得正确答案而获奖励。

对一个问题，Fugu-Ultra 写出一组 G 个不同工作流。每个被评分，然后在组内相对比较——**比组平均好的被推高，差的被压低。** 不需要独立评判网络，训练更简单便宜。

关键公式——优势：A_i = (r_i - mean(r_1,...,r_G)) / std(r_1,...,r_G)

假设 G=4 个工作流：r1=1.0（有效正确）、r2=0.5（有效错误）、r3=0.5（有效错误）、r4=0.0（格式错误）：

步骤 1 计算均值：mean=(1.0+0.5+0.5+0.0)/4=0.5。步骤 2 计算标准差：差异 0.5,0,0,-0.5 → 平方 0.25,0,0,0.25 → 平均 0.125 → sqrt=0.354。步骤 3 算每项优势：A1=(1.0-0.5)/0.354=+1.41（强化），A4=(0.0-0.5)/0.354=-1.41（抑制），A2=A3=0（中性）。

除以标准差使优势在公平尺度上。工作流 1 因正确获正优势强化；工作流 4 因格式错误获负优势抑制；两个中等的中性。经过大量问题，Fugu-Ultra 学会写出像那些有效工作流的工作流。

完整 GRPO 还有两个部分：clip 防止单步更新过大，KL 项保持新模型接近原始。论文训练 Fugu-Ultra 时没用 KL 项。

**防止 Agent 抄袭——两个问题，两个方案**

当多个模型作为 Agent 运行并随时调用工具（运行代码、搜索网络、读取文件）时，出现两个问题。

**问题 1：编排坍缩。** 如果每个 Agent 都能看到第一个 Agent 的完整历史，第一条路径变成所有人的路径——后面的跟着重复同样的工作，不带来新想法。团队坍缩为一个意见。

**解决方案：工作流内隔离。** 在一个工作流内，每个 Agent 只通过访问列表看其他 Agent 的工作，否则只看自己的过去行动。这给了每个 Agent 用自己的方式解决子任务的自由。

**问题 2：忘记一切。** 但完全隔离也不好。长时间多轮对话中 Agent 应记住已调用的工具，避免重复调用和反复发现同样的事实——例如已经搜索过某个 API 文档就不需要再做一次。

**解决方案：持久共享记忆。** 跨工作流共享过去工具调用的记忆。工作流内保持独立，跨整个对话共享有用背景。**这种平衡使团队既多样又高效。**

**Fugu 的表现——击败池中所有模型**

Fugu 和 Fugu-Ultra 在多项基准上达到最先进水平，**在许多测试中击败了自己池中的模型**：

| 基准测试 | Fugu-Ultra | Fugu | Claude Opus 4.8 | Gemini 3.1 | GPT-5.5 |
|---------|-----------|------|----------------|------------|---------|
| SWE Bench Pro（修复真实 bug） | **73.7** | 59.0 | 69.2 | 54.2 | 58.6 |
| Terminal Bench 2.1（终端使用） | **82.1** | 80.2 | 74.6 | 70.3 | 78.2 |
| GPQA Diamond（硬科学） | **95.5** | **95.5** | 92.0 | 94.3 | 93.6 |
| Humanity's Last Exam（极难推理） | **50.0** | 47.2 | 49.8 | 44.4 | 41.4 |

**最扎眼的是 SWE Bench Pro 上 Fugu-Ultra 73.7 分超过所有池中模型**——Claude Opus 4.8（69.2）、GPT-5.5（58.6）、Gemini 3.1（54.2）。通过良好组合，团队比任何单一成员都强。它还胜过了一些不在池中且未公开的模型，印证了论文的判断：编排是在训练更大模型之外的一种真实的能力获取方式。

论文还在标准基准之外做了有趣测试：

- **魔方求解器：** 每个模型要写解魔方的代码。Fugu 模型解决全部 300 个测试魔方，Fugu-Ultra 找到最短解法平均 19.72 步（已知最佳可能 20 步）。**三个前沿模型中有两个写的代码在解第一个魔方前就崩溃了**
- **盲棋：** 不显示棋盘仅文本走法，模型在脑中追踪整盘棋。Fugu 赢得全部四盘棋，包括对阵强引擎，且没有任何错误
- **在线股票交易：** 从 $10,000 开始的 50 周模拟，Fugu-Ultra 增长到 $11,943（+19.43%），高于每个单独模型

Fugu 还在自主 AI 研究、古典日文手写字母阅读、CAD 中生成可工作机械零件上表现良好——编排后的团队每次都优于单一模型。

**Fugu 自己发现的聪明策略**

**这些策略不是人写的——训练过程中自己冒出来的。** 论文观察到三种：

**辩论与聚合。** Fugu-Ultra 构建树，模型独立尝试问题后选定模型合并最佳部分——在 Conductor 部分已见过。

**构建与调试。** 编码任务中 Fugu-Ultra 通常用 GPT 作为构建者，再在关键时刻引入 Opus（调试强）找 bug 和安全漏洞，然后把发现传回去。例如构建小型包服务器时，Opus 发现 GPT 遗漏的几个具体风险，传回后 GPT 正确完成了工作。

**引入专家。** 只在需要特殊技能时拉入特定模型。密码分析任务中用 Opus（安全强）发起攻击，再用 GPT（数学强）重新推导攻击所需的确切数学。

**Fugu 学会了每个模型的细粒度优势以及如何组合它们，像一个知道在什么时候该叫谁的好经理。**

核心观点：编排是一条新的 scaling axis——当更多人堆更大模型时，让已有模型协作好可能是更实际的路径。Fugu 把模型选择从工程决策变成可学习问题，Fugu-Ultra 自发学会的反抄袭和私密通信意味着编排器自己就能优化协作策略。
