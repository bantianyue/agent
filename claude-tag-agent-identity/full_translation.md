# 完整翻译版

## 第一部分：Claude Tag 产品介绍

Claude Tag 是团队与 Claude 协作的新方式。

从 Slack 开始，Claude 可以像团队成员一样加入。授予 Claude 访问选定频道的权限，连接你选择的工具、数据甚至代码库。然后，频道中的任何人都可以 @Claude 并委派任务给它，同时专注于其他工作。Claude 通过记住所在频道的相关信息来构建上下文，并能规划未来要完成的任务。

Anthropic 将 Claude Tag 视为 Claude Code 演进的开端：它让模型更加主动，并且能更好地与整个团队协作。@Claude 现在是 Anthropic 内部完成工作的主要方式之一。目前，产品团队 65% 的代码由内部版本的 Claude Tag 创建。同样的模式正在扩展到工程之外——团队用 Claude 追踪产品指标和数据、处理支持工单，甚至帮助排查棘手 bug 的根本原因。

Claude Tag 在 Slack 上推出，因为 Slack 是团队与 AI 协作的自然场所，也是 Anthropic 日常工作的主要平台。即日起，Claude Enterprise 和 Team 客户可以体验 Beta 版。目标是扩展到更多平台。

## 使用 @Claude

如果你用过 Claude Code 或 Cowork，Claude Tag 会感觉很熟悉。用简单语言 @Claude 提出请求，它会将任务分解为多个阶段，然后依次完成，使用它能访问的工具。完成后，它会在 Slack 线程中回复创建的内容。

@Claude 的几个新优势：

**@Claude 是多玩家的。** 在 Slack 频道中，只有一个 Claude 与所有人互动。任何人都可以看到它在做什么，可以从上一个人停下的地方继续对话。这更像与队友协作，而不是单聊。

**@Claude 会随时间学习。** 随着 Claude 跟随频道内容，它会积累更多关于工作的上下文。用户不需要反复从头解释。如果被授予权限，Claude 甚至可以从其他 Slack 频道和数据源自动学习（它不会从私密频道报告）。

**@Claude 会主动行动。** 如果启用"环境感知"行为，Claude 会主动更新你可能需要知道的信息。它会标记来自各个频道和连接工具的相关信息，跟进已沉寂但未解决的线程或任务。

**@Claude 异步工作。** 给 Claude 设置一个任务，你可以专注于其他优先事项。它还可以为自己安排任务，在数小时或数天内自主推进项目。

你也可以给 Claude 发送私信：它会私下回复，使用你设置的个人工具和连接器。

## 上手步骤

系统管理员指定模型应该访问哪些工具和信息、在哪些频道中。可以理解为为不同用途创建独立的 Claude 身份：所有内容，包括记忆，都将限定在管理员定义的频道范围内。例如，为销售工作设置的模型不会将记忆传递给为工程设置的模型。

权限设置好后，每个人都可以立即开始 @Claude。管理员可以设置 token 消耗限制（组织和单个频道），并查看 @Claude 所做一切的日志。

如果你是 Claude Enterprise 或 Team 客户，即日起可体验 Beta 版。四个步骤：
1. 将 Claude Tag 与 Slack 工作区配对
2. 授予 Claude 工具访问权限
3. 设置月度消费上限
4. 在私密频道中测试确认工作正常

Claude Tag 取代了现有的 Claude in Slack 应用。Claude Tag 运行在 Opus 4.8 上。

## 第二部分：Agent Identity 访问模型

要让 AI Agent 在人与 Agent 组成的团队中发挥最佳作用，它需要访问人类拥有的相同工具、文档和上下文。

在"单人"AI 体验中（一个人与一个助手对话），这很简单：你连接自己的账户，Agent 代表你行动。但在"多人"AI 体验中（如 Claude Tag），Claude 与多人同时在一个共享频道中，它使用的是属于工作区的工具和上下文，而非任何个人。

要让多人体验工作，Claude 需要拥有这些工具的自己账户，由管理员设置并与工作区绑定。这种访问模型称为 Agent Identity（Agent 身份）。

### 为什么"以用户身份行动"会失效

用 AI 作为个人助手时，你可以连接 Google Drive、GitHub 和日历等平台，让模型用你的权限读写。但这对 Claude Tag 不适用，有两个原因：

**Agent 自主性在增长。** AI Agent 能独立可靠完成的任务时长大约每四个月翻一番。Agent 现在可以为自己安排未来的任务，在请求者下线后很久仍然响应事件。用户设置触发条件让 Agent 在特定情况下行动，但 Agent 基本是自主工作的。

**多人团队。** Claude Tag 将 Claude 放在团队已经在工作的共享空间中——例如，三个工程师和一个 PM 一起调试的频道。但超过一个人指挥时，该用谁的权限？没有哪个单一选择在所有时候都是正确的。这让管理员可以定义 Agent 在 Slack 中能做什么，独立于涉及的人类用户，并独立追踪 Agent 在 Slack 中的操作。

### Claude 以自身身份行动

在启用了 Claude Tag 的频道中，Claude 不代表任何单个用户行动。它在每个接触的系统中都有自己的账户：以 Claude 应用的身份在 Slack 中发帖，以 Claude GitHub App 的身份创建 PR，以管理员配置的服务账户查询数据仓库。

因为没有个人用户凭证参与，共享频道永远不会成为进入某人私密文档的后门。

### 权限继承

在 Agent Identity 模型中，管理员在工作区级别定义一个身份——Claude 在所有地方持有的连接和技能基线——每个频道默认继承。然后，在需要的地方，可以在频道级别覆盖，例如授予工程频道访问 GitHub 和数据仓库的权限，或将 CRM 连接限制在单个私密频道。

除了凭证，管理员还定义：
- **仓库访问：** Claude 可以读写哪些仓库
- **连接器：** Claude 用来完成工作的工具和 API 密钥。不同 API 密钥可以以不同权限级别连接到同一服务
- **技能和插件：** Claude 动态加载的指令、脚本和资源文件夹
- **常设指令：** 每个频道的自定义指令和上下文

因为模型围绕不同的 Claude 身份工作，撤销身份就结束了 Claude 在所有使用该身份的地方的访问权限。

### 身份边界如何工作

Claude Tag 为每个私密频道创建独立的身份；工作区中的公开频道共享一个工作区级身份。Claude 在法律频道的身份无法访问未授权的代码，在工程频道的身份无法读取未授权的法律文档。记忆和访问都遵循这些边界：Claude 在私密频道中学到的东西不会出现在更广泛的工作区中。

身份属于频道，所以频道中的任何人都可以默认 @Claude。在 Enterprise 计划中，基于角色的访问控制让管理员可以决定哪些成员可以调用 Claude。

### 工具和上下文的广泛默认访问

Claude Tag 在 Anthropic 内部运行时发现，其价值随着工具和上下文访问的增加而复合增长。每个连接的系统让其他每个系统更有用，因为 Claude 可以跨系统组合上下文。

获得最多价值的团队是那些从一开始就授予 Claude 广泛访问权限的团队，然后根据组织偏好逐步收窄。建议：在几个频道中以基线 profile 开始，阅读审计追踪，然后在工作证明合理时扩展访问权限。

### 私信

在 Claude Tag 中，私信的工作方式与共享频道不同。私信使用用户个人的 claude.ai 账户——他们的连接器、凭证和结果上的名字。这使得私信成为处理不应出现在频道中的任务和工具的正确场所。

### 安全与审计

当管理员向频道 profile 添加连接时，凭证独立存储并映射到该频道的身份，然后在请求时在网络边界注入。流向管理员未允许主机的出站流量被直接阻止。审计方面，每次 routine、记忆写入和使用 Agent 凭证的网络调用都被记录。

### 未来方向

Agent Identity 是 Claude Tag 访问模型的基础。未来计划包括即时凭证授权（用户可以在不永久扩大 Agent 范围的情况下批准单个敏感操作），以及面向更复杂安全等级组织的身份感知覆盖层。
