ChatGPT remembers you, but not the way most people assume.

It does not search your old conversations. There is no vector database quietly pulling up your history when you ask a question. Instead, on every message, ChatGPT loads a summary of who you are, built from your past chats, straight into its context.

In June 2026, OpenAI rebuilt how that summary gets made and gave it a name: Dreaming.

This article breaks down how ChatGPT memory actually works: the two systems OpenAI documents, the injection mechanism researchers reverse-engineered, what Dreaming changes, the risks that come with a permanent profile, and where a developer memory layer like Mem0 fits.

The two systems OpenAI documents

Officially, ChatGPT memory has two parts (OpenAI Help Center).

Saved memories are discrete facts. You can tell ChatGPT "remember that I am vegetarian," or it can save a detail on its own when it seems useful later. OpenAI describes these as working like custom instructions, except the model updates them automatically instead of you managing them by hand.

Reference chat history is different. When it is on, ChatGPT draws on your past conversations to infer your interests and preferences, and uses that to personalize future chats. Unlike saved memories, which persist until you delete them, these inferred details shift over time as the model updates what it thinks is worth keeping (OpenAI).

You can turn either off in settings, edit what ChatGPT knows about you in conversation, or use Temporary Chat for sessions that neither read nor write memory. Custom GPTs do not share this memory; each runs fresh.

That is the official picture. It tells you what happens. It does not tell you how.

How it actually works: a profile, not a search

The surprising part, and the part OpenAI does not spell out, is the retrieval mechanism. There isn't one.

Security researcher Johann Rehberger tested this directly and found ChatGPT could not recall specific one-off conversations from a year back. His conclusion: "ChatGPT actually cannot search through your history at the moment" (Embrace The Red).

Simon Willison reached the same place, retracting his own initial guess that it used retrieval-augmented generation: "It looks like that's not the case." What actually happens, in his words, is simpler: "ChatGPT effectively maintains a detailed summary of your previous conversations, updating it frequently with new details. The summary then gets injected into the context every time you start a new chat" (simonwillison.net)

So memory is not fetched per question. It is pre-loaded. A standing profile of you rides along in the system prompt on every turn. Rehberger documented that this profile arrives in roughly six named sections: Model Set Context, Assistant Response Preferences, Notable Past Conversation Topic Highlights, Helpful User Insights, Recent Conversation Content, and User Interaction Metadata. Willison independently confirmed four of those section names in his own extraction.

The detail is what makes it concrete. Recent Conversation Content holds your recent chats, but only the messages you typed, not ChatGPT's replies. Rehberger's read on why: "otherwise it would be too much data and prompt injection and hallucinations would become a significant risk." Entries are split by a |||| delimiter, tagged with internal intent_tags like translation or argument_or_summary_generation, and timestamped, down to the second for the most recent few, then only to the hour for older ones.

Saved memories work through a different path: a bio tool the model calls (to=bio) to persist a fact, which then shows up in the # Model Set Context section of future system prompts as a dated entry (Embrace The Red).

One caveat on all of this. The mechanism above was mapped by probing ChatGPT through 2024 and 2025. Dreaming, covered next, changed how the profile is built, and no one outside OpenAI has published a look at the post-update internals. So treat this as the most recent independent picture of how memory is read, not a guarantee of today's exact wiring.

The bitter lesson: include everything, every time

Why build it this way instead of the careful retrieval pipeline most memory systems use? The writer behind shloked.com framed it as OpenAI betting on scale over cleverness: "No extraction of individual memories. No vector databases. No knowledge graphs. No RAG." Instead, "OpenAI just includes everything with every message. Your compressed User Knowledge Memories, Model Set Context, recent conversations, metadata. All of it, every time" (shloked.com).

The logic is that a strong enough model with a big enough context window does not need a retrieval layer deciding what to surface. Hand it the whole profile and let attention sort it out. It is simple and it scales to a consumer product with hundreds of millions of users. It also has a ceiling, which is exactly the problem Dreaming was built to solve.

Dreaming: memory that updates itself

A pre-loaded profile rots. Facts pile up, go stale, and contradict each other, and a hand-curated list of saved memories does not scale to years of chats across hundreds of millions of people. So in June 2026, OpenAI shipped a new way to build the profile.

In its announcement, titled "Dreaming: Better memory for a more helpful ChatGPT," OpenAI described it as "a more capable and scalable system for synthesizing memory, developed to tackle the staleness, correctness, and scalability challenges that we observe when memory is applied to the hundreds of millions of users and multi-year time horizons in ChatGPT" (OpenAI).

In plain terms: Dreaming is a background process that reads across years of your conversations and rewrites what ChatGPT remembers about you, without you asking it to save anything. It began rolling out on June 4, 2026, to Plus and Pro users in the US first (Tech Times). OpenAI's canonical example is a memory revising itself: "You're going to Singapore in July" becomes "You went to Singapore in July 2026" once the trip is over. No user action.

OpenAI reports the change improved its internal evaluation of factual recall from 41.5% in 2024 to 82.8% with the 2026 architecture, with preference adherence at 71.3% and time-sensitive accuracy at 75.1%. Those are OpenAI's own numbers, on an evaluation it has not released, and no independent party has reproduced them, so treat them as vendor-stated, not audited.

The name is the tell. ChatGPT now consolidates memory in the background, between sessions, the way the rest of the field has started to. The same week's research and products reached for the identical metaphor: Google's "Language Models Need Sleep" coined a Dreaming phase for weights, and memory products from Mem0 to GBrain ship idle-time consolidation. OpenAI putting "Dreaming" on the consumer product is the loudest signal yet that background consolidation, not bigger context, is where memory is heading.

The cost of a permanent profile

A standing, human-readable profile of you is powerful, and that is also the problem.

Willison, who is not alarmist by temperament, called it a privacy concern worth pausing on: ChatGPT is "effectively collecting a dossier on our previous interactions, and applying that information to every future chat." His sharper question: "Has there ever been a consumer product that's this capable of building up a human-readable profile of its users?" His objection is not that memory exists, but that the profile is "dumped into the model's context without my explicit permission or control."

There is a security edge too. Because untrusted content can reach the model, it can also reach the memory.

Rehberger demonstrated that a Google Doc referenced in a chat could write attacker-controlled entries into long-term memory: "All the instructions we had in our external document were followed and each bullet point became a memory" (Embrace The Red).

A 2024 study showed the feature could be turned into a standing surveillance channel, with the model instructed to watch for and store a user's personal data for later exfiltration (arXiv:2406.00199). These were proof-of-concept attacks, and OpenAI has repeatedly hardened the exfiltration paths since, so do not read them as live exploits today.

The durable lesson is structural: memory is a write surface, and anything that can write to it is part of your attack surface.

Where it stops

ChatGPT's memory is built for one thing extremely well: making one consumer app feel like it knows you. The design choices that make it good at that are also its limits.

It is closed and single-app. The profile lives inside ChatGPT and follows you nowhere else. Your editor agent, your terminal agent, and your own application cannot read or write it. It is pre-loaded, not retrieved, so you do not get to decide which memory surfaces for a given task; you get the whole dossier or nothing. And it is built for end users, not developers: there is no clean way to scope it, branch it per agent, or carry one identity's memory across the tools that person actually uses.

Where a developer layer fits

That last gap is the one a dedicated memory layer like Mem0 is built for. The contrast is not consumer versus enterprise; it is a closed profile versus a retrieval system you control.

Three differences matter.

1. Mem0 retrieves instead of pre-loading: its stack runs three scoring passes in parallel, semantic similarity, keyword matching, and entity matching, and fuses them, so the memory that surfaces is the one that fits the query rather than the entire profile every time.

2. Mem0 reports this runs at under 7,000 tokens per call versus 25,000-plus for stuffing full context, at comparable accuracy (these are Mem0's benchmark figures).

3. And memory is identity-scoped and portable: every write is tagged with identifiers like user_id, agent_id, and run_id that compose at retrieval time, so one person's memory follows them across apps, agents, and sessions instead of living inside a single product.

ChatGPT proved that a profile injected into context is enough to make an app feel personal.

The harder version of the problem, memory that you retrieve deliberately, scope precisely, and carry across every tool, is where a dedicated layer earns its place.
