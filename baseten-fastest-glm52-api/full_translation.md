# 全文翻译 — How we built the world's fastest API for GLM-5.2

翻译规则：原文每一句话都要有对应的中文翻译，不添加任何分析评论，不排版，代码块保持原文不变。

GLM-5.2 是自 DeepSeek-R1 以来开源模型领域最大的新闻。

原因显而易见。GLM-5.2 在纯 token 计价上通常比 GPT 5.5 和 Opus 4.8 便宜 70-80%（可以用我们的计算器估算你工作负载的节省金额），同时性能相当。

但一个模型仅仅聪明和便宜是不够的。要在生产环境中发挥作用，模型必须快速、可靠、可大规模可用。兑现前沿开放智能的承诺，需要卓越的推理性能。

因此，我们构建了世界上最快的 GLM-5.2 API，目前由 Artificial Analysis 测得每秒超过 280 个 token。

我们通过在整个推理过程中利用多种技术实现了这一性能：
- 更新我们的定制推理引擎，为 GLM-5.2 架构实现共享 DSA
- 从原始 FP8 权重运行并校准自研 NVFP4 量化，在 BFCL 等 Agent 基准测试上表现同等质量
- 通过基于 NVIDIA Dynamo 工具构建的 KV 感知路由实现高 KV 缓存命中率，降低预填充负载并改善重复前缀请求的 TTFT
- 通过使用 NVIDIA Dynamo 工具包构建的分离式推理，在观察到的负载形态上实现 2 倍更高的 TPS
- 通过实现对 GLM-5.2 多 Token 预测（MTP）头的支持，进一步提升 TPS

你可以通过 Baseten Model APIs 亲自体验 GLM-5.2 的性能。我们也为高负载工作负载提供 GLM-5.2 专用部署。

## GLM-5.2 概览

Z.ai 的 GLM-5.2 是一个 744B 参数的前沿 LLM，擅长 Agent 任务（尤其是编码），支持高达 100 万 token 的上下文窗口。它采用了与其前身 GLM-5.1 相似的架构：混合专家模型（40B 活跃参数）、非思考和思考模式，以及完全开放的 MIT 许可证。虽然 GLM-5.2 与 GLM-5.1 有很多共同点，但它现在使用了共享 DSA 权重，我们在定制运行时引擎中实现了对此的支持。

GLM-5.2 有出色的基准测试分数，但到如今 AI 构建者都知道，一个模型的实用性不止于标准评测上的表现。在实践中，GLM-5.2 达到或超过了其基准测试所暗示的能力。它确实是一个出色的模型，适合编写代码、运行 Agent 以及其他前沿语言模型任务。

## Blackwell GPU 的高质量 NVFP4 量化

我们在 NVIDIA Blackwell GPU 上运行模型 API，使用 Baseten Inference Stack 内的定制推理引擎。选定运行时使用 NVFP4 权重以获得最大性能。从原始 FP8 权重出发，我们使用 NVIDIA ModelOpt 进行了自研 NVFP4 量化。NVFP4 是 NVIDIA 的一种 4 位浮点数据格式，使用双缩放因子来保持高动态范围并保留模型质量。

在对量化模型进行校准和测试时，我们专注于确保 GLM-5.2 在 Agent 的常见模式上表现可靠。在 BFCL 函数调用基准测试上，我们观察到原生 FP8 权重与 NVFP4 量化之间的性能大致相当，多次运行的分数都在基准测试的误差范围内。

NVFP4 量化通过解锁更快的 tensor core 并减少 VRAM 带宽负担，从而改善了首次 token 时间和每秒 token 数。

## 使用 NVIDIA Dynamo 进行缓存感知路由

GLM-5.2 特别适合长上下文请求和复杂的 Agent 任务。这些工作负载通常有非常长的输入序列。通过在请求之间复用 KV 缓存，我们可以跳过共享序列的昂贵预填充过程。

我们通常从首次 token 时间（TTFT）的角度讨论 KV 缓存复用。然而，像 GLM-5.2 这样的推理模型更关心首次回答 token 时间（TTFAT），它结合了 TTFT 与推理序列的部分 TPS。

这张图表显示，在生成首个回答 token 的平均 7.9 秒中，有 7.1 秒用于生成推理 token，而只有 0.8 秒用于处理输入序列。

尽管如此，将 TTFT 降到 800 毫秒对系统的整体响应能力和吞吐量仍然很重要。在大规模生产部署中，KV 缓存在各个独立副本之间分布。我们使用 NVIDIA Dynamo 的工具来路由传入请求。

在多租户 API 上，精确的缓存命中率取决于任意时刻的具体流量特征。到目前为止，我们在相当异构的流量中观察到了高命中率，这降低了预填充的负载并改善了端到端性能。

## 使用 NVIDIA Dynamo 进行预填充-解码分离

我们对性能影响最大的优化之一，是对 GLM-5.2 进行预填充和解码的分离。

LLM 推理有两个不同的阶段：

- **预填充（Prefill）**：处理输入序列、构建 KV 缓存并生成第一个输出 token 的计算密集型过程。预填充性能决定 TTFT。

- **解码（Decode）**：生成后续输出 token 的内存密集型过程。解码性能决定 TPS。

传统上，单个 GPU 节点同时处理预填充和解码。有了分离式架构后，这些工作负载在独立的引擎上运行。这带来了几个好处：

- 预填充和解码独立运行，不会争抢资源
- 我们可以根据需要为预填充和解码分配不均等的资源（通常，我们配置的预填充引擎多于解码引擎）
- 运行预填充和解码的推理引擎可以采用不同的配置，针对各自推理管道的特定需求进行优化

KV 缓存仍然尽可能复用，这意味着预填充工作节点只用于处理新的输入序列。

实现 PD 分离的大部分挑战在于预填充引擎和解码引擎之间可靠、低开销的通信和编排。NVIDIA Dynamo 提供了一个开发者工具包，用于实现分离的关键组件：

- 一个预填充队列，在所有预填充引擎饱和时持有请求
- 对条件分离的稳健支持，基于可配置的输入序列长度阈值（在前缀缓存和预填充队列大小之后）进行预填充路由
- 基于 NIXL 的高效 KV 传输，从预填充引擎到解码引擎，并有一个内核用于在引擎具有不同 TP 配置时在布局之间转置 KV 块

在 GLM-5.2 的聚合部署与分离部署的头对头基准测试中，我们观察到分离式推理的每秒 token 数高出 2 倍。

## 通过多 Token 预测实现更高的 TPS

GLM-5.2 配备了一个改进的多 Token 预测（MTP）层，降低了生成 draft token 的成本并提高了这些 token 的接受率。

回顾一下，MTP 是推测性生成的几种方法之一。推测性生成是指在前向传播中一次生成多个 token，目的是提高 TPS。由于所有算法都有验证步骤，推测性方法是无损的性能优化。

利用这些 MTP 层生成 draft token，我们测试了各种序列长度，以找到生成长序列和保持高接受率之间的最佳平衡。我们在过去几个月对 MTP 做了大量工作，GLM-5.2 使用的推测性生成仍有提升空间。

看到这类基准测试结果，自然想问：同样的性能在生产环境中能否保持？

事实上，我们不仅能在生产环境中提供这一性能，还能为 GLM-5.2 的大规模专用部署实现更好的工作负载特定性能。可用的杠杆包括：
