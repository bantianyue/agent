2026 年 2 月，Nous Research 发布了 Hermes Agent——一个不活在 IDE 里、关标签页不会失忆的开源自托管 Agent。它在你的机器上作为守护进程运行，跨会话保持记忆，用自然语言接受定时任务，从经验中自写可复用技能。

作者 Mnimiy 在自己的 $5 VPS 上跑了 5 周，用 Claude 做底层模型。结论是：**空白安装什么也不做。它是运行时，不是工作流。** 把它从"star 的仓库"变成"半夜做了我 3 小时工作"的东西，是第一天给的 prompt。而几乎没人分享这些 prompt。

这篇就是他的 17 个 prompt 和配置。

---

## 1 为什么需要持久 Agent

几乎所有 AI 工具都是会话式的。Claude Code、Cursor、聊天窗口——打开、工作、关闭、上下文随会话死亡。这对单项目编码是对的。

但有一整类工作不是会话形状的：你醒来前应该准备好的简报、你在不在都在被监视的构建、你埋头时一直在填充的收件箱。会话工具接不住，因为它们在该运行的时候没在跑。

这就是 Hermes 填补的缺口。持久化、可定时、可触达。

---

## 2 三个前置条件

三个条件缺一不可：

**模型要有真上下文。** 小模型在多步骤任务中途丢掉工具调用。前沿模型有足够余量。Claude 轻松胜任。

**后端不闲置计费。** 无服务器后端在任务间隙休眠，24/7 Agent 不等于 24/7 账单。

**持久化宿主机。** 笔记本不行。$5 VPS 可以——合上盖子不能杀死凌晨 3 点的工作。

---

## 3 17 个 prompt

**1. 晨间简报**
> 每个工作日早上 7 点，拉取我未读的 GitHub 通知和开放的 PR，总结什么变了、什么被阻塞了，以 3-5 条要点发到 Telegram。

作者说：以前每天早上花 35 分钟手动翻 6 个仓库，每周约 3 小时。现在坐下前简报已在 Telegram 等着。

**2. 仓库监视**
> 监视 [org/repo]。保持静默，除非 CI 变红或有新 issue 带标签 "bug"。触发时发失败 job 名称或 issue 内容，别的不用发。

**3. 收件箱分类**
> 每小时检查我连接的频道，按发件人和紧急程度分组，自动归档新闻通讯，只升级提到截止日期、有人在等我、或钱的条目。

作者说：约 120 条/天跨 6 个平台，这条升级规则是全部 prompt。

**4. 研究摘要**
> 每星期五下午 6 点，搜索 [主题] 的新发布和严肃讨论，与上周去重，以 5 条要点+链接发到 Telegram。

去重子句是诀窍：它记得上周，所以落地的内容是真的新的。

**5. 仓库冷启动**
> 克隆 [repo URL]，用 5 条要点总结架构，找到主入口和最危险的文件，起草干净的 PR 工作流。

**6. 异步研究**
> 研究 [问题]，比较前 3 个选项的价格、限制和锁定，今晚完成后发结果。不要等我跟进，做合理假设并在顶部列出。

**7. 竞品监视**
> 每天早 9 点检查 [产品 A]、[产品 B] 的更新日志和定价页，只有真正变了的时候发消息。

**8. 夜间代码审查**
> 每晚 11 点检查今天的 commits，标记任何风险：留下的 TODO、发出去的 console.log、超过 80 行的函数、无测试的改动路径。

作者说：把一个带 token 的 console.log 发出去了一周才发现。夜间审查现在在那类事发生前就抓住它。

**9. 站会**
> 每个工作日 9:55，从仓库和频道整理站会内容：昨天完成了什么、进行中、被阻塞，三条要点。

**10. 提及雷达**
> 每天一次，搜索 [项目名] 在网页和平台上的新提及，忽略表扬，升级 bug 报告、投诉和未回答的问题。

**11. 视频转要点**
> 取 [视频/播客 URL]，拉取转录，用 5 条要点+时间戳给出论点，跳过片头和赞助。

**12. 解释这个报错**
> 有栈追踪：[粘贴]。在仓库中搜索原因，用两句话解释实际失败原因，起草最小的修复 patch。

作者说：一个生产栈追踪通常花一小时 bisect。现在它给我失败行、两句话原因、三行 patch——我还在读追踪它就回来了。

**13. 收件箱零草稿**
> 对常规邮件（安排会面、自我介绍、状态更新）用我的语气起草回复，放入队列等批准。绝不自己发送。

**14. 值班诊断**
> 当监控告警触发时，拉取相关日志最后 50 行，检查最近部署了什么，发一段落的原因初步猜测。

**15. 配置模型**
```bash
hermes config set model anthropic/claude-opus-4.8
```

**16. 配置后端**
```bash
hermes config set terminal.backend daytona
```

**17. 保存为 skill**
> 成功了。把它保存为可复用的 skill "morning-brief"，这样下次不用重新解释格式。

作者说：第四遍重新解释晨间简报的格式后才想到让它永久化。Hermes 从这次运行中自写 SKILL.md，之后"运行晨间简报"就懂了。

## 4 三个常见错误

**模糊的时间表。** "给我的仓库发更新"产生数据洪流。每个定时 prompt 现在都带"只在 X 时告诉我"子句。

**小时级任务无 token 预算。** 一个话痨的小时级 triage 一周的支出超过了整个月的预算。持久化 + 无上限 = 惊喜账单。

**用便宜模型省钱。** 小模型在任务中途丢掉工具调用，失败方式看起来像 prompt bug。模型不是省钱的地方。

---

## 5 一周对比

作者记录了同一个工作周两次——一次手动，一次交给 Hermes。

无人值守 = 按计划或异步运行，初始 prompt 后无需输入。

关键不是省了几小时，而是这些工作没有一件发生在他必须醒着的时候。稀缺资源从来不是时间，是注意力。持久化 Agent 的意义在于工作不再竞争注意力。

---

## 6 一点观察

**这是目前看到最实用的 Hermes prompt 集合。** 不是理论，是跑在 $5 VPS 上、每天实际干活的东西。每条 prompt 都有触发条件、具体操作和升级规则，缺一不可。

**"提示词即岗位描述"的视角转换很关键。** 给聊天窗口的 prompt 是一个问题。给持久化 Agent 的 prompt 是一个岗位描述——需要触发器（时间表或事件）、正文（做什么）和升级规则（什么时候打扰你）。丢掉任何一个，prompt 要么从不触发，要么做错事，要么用噪音淹死你。

**"保存为 skill"的复利效应被低估。** 每条成功运行的 prompt 变成永久能力，下次只需说"运行 morning brief"。7 天 vs 30 天的差距不是线性增长——是 skill 库从 0 到 15+ 的拐点。

**作者诚实列出了错误，这比成功案例更有价值。** 模糊时间表、无 token 预算、用小模型——这是新手三连坑。特别是模型选择：他在便宜模型上 debug 了 2 周才发现所有 prompt 都正确，是模型撑不住多步骤任务。这条经验值 $5 VPS 多周的时间成本。
