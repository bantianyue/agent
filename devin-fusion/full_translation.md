# Devin Fusion：以 35% 更低成本实现前沿性能

Cognition 团队，2026.06.29

工程团队正在把钱烧掉。对每个任务都使用最昂贵的模型已经不再可持续。但现有的模型混用工具很糟糕——它们在大多数基准测试上看起来不错，但在实际写代码时却一塌糊涂。

Cognition 专注于在不牺牲智能的前提下跨前沿模型进行路由。今天，我们分享一种新型多模型编排框架 **Devin Fusion** 的工作，它在混合模型方面显著更好，同时降低成本并在真实使用中保持智能。我们发现它在 FrontierCode（一个衡量代码正确性和质量的最新编码基准）上保持了 **前沿和 Fable 5 级别的性能，成本降低 35%**。

在本文余下部分，我们将分析为什么好的模型路由如此困难，以及使一切工作的两个关键技术：**"Sidekick"方法和动态会话中路由**。

## 诀窍：Sidekick

我们架构的核心思想是运行两个并行的 Agent：一个使用前沿模型，另一个使用更具成本效益的"Sidekick"模型。两者都是功能完备的 Agent，拥有自己的工具集和收集/操作上下文的能力。

随着任务推进，主 Agent 决定哪些任务交给 Sidekick，哪些自己做。然而，让 Sidekick 在实际中良好工作需要深度调整交互模式。我们发现主 Agent 应该尽可能少操作，只读取绝对必要的内容。默认情况下它应该委派和监控，同时做出重大决策：计划、歧义解释、最终审查。

这种方法解决了更基础模型路由的主要问题：
- **它保留了真正的前沿智能，而不是"基准测试分数"智能。** 路由器经常过拟合特定基准。通过保留前沿模型在组合中，Sidekick 方法继续受益于前沿模型的创造力和通用智能。
- **它能泛化到单提示任务和问答之外。** 模型路由器通常为整个任务路由到单个模型。提示通常不包含足够的信息来正确判断难度。而且用户可能对简单的初始提示提出困难的后续问题。能够在聪明模型和 Sidekick 之间动态切换使系统更加稳健。
- **它避免了模型间路由时昂贵的缓存未命中。** 我们之前探索过"Smart Friend"工具，Anthropic 也发布了类似的"Advisor"工具。这两个想法的核心是让一个模型拥有一个工具来查询另一个模型获取有用建议。但问题在于：每次调用另一个模型时，任务的上下文没有以可缓存的方式共享，你需要付出非常昂贵的代价。在 Sidekick 设置中，主模型和 Sidekick 模型都维护自己持久的、可缓存的上下文。

当然，为了实现 Devin Fusion 的能力，我们克服了许多实现细节。例如，大多数缓存输入只有 5 分钟的有效期。我们鼓励读者思考如何围绕这个进行工程设计。我们很乐意交流笔记。

### Sidekick 随着模型变聪明而扩展得更好

最近的模型，特别是 Fable 5，在这些多 Agent 设置中表现异常出色。Fable 更智能地委派工作、更高效地请求上下文、更精确地规划，所有这些都带来了更大的成本改进，同时对智能的影响极小。这表明 Sidekick 模式是一种随着基础模型变好而变得更加有用的模式。

在我们的测试中，Fusion + Fable 5 比纯 Fable 5 编排便宜 41%，而 Opus 和 GPT-5.5 级别的模型为 35%。这个差距看起来可能不大，但我们认为它低估了真正的差异。非 Fable 的数字反映了多轮 Devin Fusion 编排的调优；Fable 5 的数字没有，因为在我们可以应用调优之前访问就被切断了。

## 动态会话中路由

有了 Sidekick，你仍然需要为任务选择合适的模型。我们根据任务类型和复杂程度为主 Agent 或 Sidekick 选择不同的模型。但如果在开始时选择一个模型，后来发现另一个更合适，那就危险了。同样，如果任务对 Sidekick 来说太具挑战性，你可能希望将其从 Sidekick 移回主 Agent。为处理这些情况，我们在任务执行期间使用轻量级分类器来指示何时需要切换回主 Agent 或完全使用不同的模型。

我们希望模型切换时实现缓存高效，这需要一些巧妙的工程。我们通过在上下文压缩期间切换模型来实现这一点——因为压缩无论如何都会触发缓存未命中。每次触发压缩时，我们利用这个机会评估情况并切换负责的模型，从而有效实现模型切换"免费"。注意这意味着我们甚至可以"升级"Sidekick 模型而不回到主模型，且没有额外的缓存惩罚。

## 结果

我们在有无使用 Fable 5 能力的情况下对我们的新编排进行了基准测试，两种配置都取得了令人兴奋的改进。

排除 Fable 5，我们的 Devin Fusion 多模型编排相对于 GPT-5.5 和 Opus 4.8 等前沿模型，在 FrontierCode 上实现了 **35% 的成本改进**，同时保持了匹配前沿的性能。

Fable 5 在这个多模型编排中表现出色，**实现了 41% 的成本降低**，同时**保持了与传统 Agent 编排中 Fable 5 相同的性能**。虽然 Fable 目前不是普遍可用的模型，但我们很高兴一旦访问恢复后，将 Fable 扩展到 Devin Fusion 的用户。

### 合理性检查

我们的目标是构建一个不仅在基准测试上表现良好，而且在实际使用中感觉良好的编排。我们为 Cognition 内部的一组用户启用了 Fusion，发现 **88% 的已合并 PR 完全由自动化 Fusion 路由器驱动**。当然，我们的最终愿望是在比内部使用涵盖的更广泛任务集上测试这个编排。这就是为什么我们非常兴奋地向云 Agent 用户发布 Devin Fusion 预览版。

## 混合模型编排日益增长的重要性

**使用单一模型完成所有工作的时代即将结束。** 前沿智能不断上升的成本在大小工程组织中都[达到了令人望而却步的水平](https://www.nytimes.com/2026/06/18/technology/ai-token-minimizing.html)。此外，现在有越来越多的不同价格和智能水平的模型选项，而且通过正确的提示，许多次前沿模型完全能够胜任大多数工程工作。你不会开兰博基尼去杂货店，那为什么要用一个能发现软件零日漏洞的模型去做圆角按钮这类小事呢？

此外，使用多模型编排可以**捕捉各种前沿模型的相对优势**。例如，在 Cognition，我们发现有些模型特别擅长 UI 测试，而不同模型擅长在 PR 中识别复杂 bug。还有越来越多[有能力的开源模型](https://devin.ai/blog/kimi-k27-glm-52-devin-desktop)。这使得训练特定领域的专门智能变得更容易。而且随着擅长特定语言、任务或库的模型出现，投资多模型能力只会变得更加重要。
