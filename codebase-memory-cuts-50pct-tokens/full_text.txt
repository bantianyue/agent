# How Codebase Memory cuts 50% token for my Claude Code

By Jason Zhou (@jasonzhou1993)

You have probably seen a dozen of these "index your codebase" tools by now. But most of them never got real adoption from me and my team - Either index got outdated OR agent forgot to call those special MCP tools;

However Codebase Memory MCP feels like something truly different; it index my whole codebase in seconds with pure programmatic building of graph + the smart design pattern of hook usage made it directly enhance the grep tool;

From the past week, it actually cuts token usage by ~50% for some tasks, and most importantly I feel more confident as my agent is well informed of the blast radius.

## The grep-and-read pattern, and where it breaks

1~2 years ago the popular bet was embeddings - index your codebase into vectors, retrieve the chunks that look related to the question. Cursor's codebase index worked this way, and a lot of MCP tools copied it.

But in real usage semantic retrieval just isn't reliable; it pulls chunks that look similar but miss the actual one you need, and it can't follow a function call from one file into another. So most teams ended up back on the simpler pattern that just works: grep, then read. It's what Claude Code leans on.

The catch is grep + read falls apart once the codebase gets big. You ask where a function is used, grep returns a wall of matches, and the agent opens them one by one - each file dumping a few hundred lines into context. In one repo that's wasteful; across the 3-4 repos a real product spans, it can't even connect them, so it guesses or stops early and misses call sites.

## Map the codebase in seconds

But the relationships are already sitting in your code. An import says one file depends on another; a function call says who reaches whom; a route says which handler answers which request. Your codebase is already a graph - grep just reads those edges as flat text and discards the structure.

Give that structure to the agent as context, and it stops reading 40 files to answer one question.

The way it builds this is pure C + tree-sitter, with no model anywhere in the loop. And that's exactly why it holds up where the embedding tools fell apart.

It's stupid fast - indexes the Linux kernel (28M lines) in ~3 min, and a normal repo in seconds. And because there's no LLM regeneration cost, the index just stays in sync with your code instead of drifting outdated.

## Give your agent a map + Smart hook design pattern

After index, agent will be given a proper toolkit that maps to how it actually works:

- get_architecture - a one-call overview of the codebase structure.
- search_graph - locate the exact node for a function, class, or route, by name or by meaning.
- trace_path - map the call chain: who calls this, and what it touches.
- query_graph - ask the graph anything in Cypher. My favorite: of all the functions that call handleOrder, which ones don't have test coverage yet - good luck doing that with grep.
- get_code_snippet - pull the exact source for a symbol.
- detect_changes - map a diff to the architecture during PR review, so the agent sees the blast radius of a change.

## The part I like most: the hook

It cater for situation where coding agent like Claude code or CodeX will forget to call special MCP tool but fall back to use grep/glob;

Instead of fighting the behaviour, It installs a pre-tool-use hook on grep - so when the agent greps a function, the grep runs as normal AND in the same step the hook looks that query up in the graph and folds the structural answer into the result. The agent never had to remember a special tool; it greps, and the graph rides along.

In some of my own test, it can save ~50% token consumptions because agent gets a proper map of the codebase rather than grepping blindly.
